# Spec: Company Linkage & Tiered Signal Expiry

## Overview
Extends the enrichment stage of the roadmap (capture â†’ extraction â†’ **enrichment** â†’ scoring â†’ review/export) with two related pieces of `CLAUDE.md`'s data model that are fully unbuilt today: (1) `Company.linked_org_id` â€” tagging a scanned/enriched `Company` as itself being a registered DASHR `Organization`, so enrichment can prefer that org's own declared product-line data over a generic scraped guess; and (2) splitting `CompanySignals` freshness into two independent TTL clocks â€” `factual_fetched_at` (6 months: registry, compliance, revenue band, website-derived product lines) and `dynamic_fetched_at` (3 months: LinkedIn firmographics, growth/momentum/news signals, Google rating, the whole IndiaMART block) â€” so a refresh only re-fetches whichever half has actually gone stale, never the whole row on one schedule.

Today `07-data-enrichment`'s `enrich_company_task`/`run_all_signal_lookups` is a strict one-shot: a `Company` is enriched exactly once (`"pending"` â†’ `"enriched"`/`"not_found"`) and never revisited, and there is no concept anywhere in the codebase of a prospect company also being a DASHR customer. This spec adds a bounded, explicitly-triggered refresh path on top of the existing one-shot flow (the CTA becomes available again once a tier goes stale, never automatic/schedule
d â€” enrichment stays a user-initiated action end to end, matching the existing "Parse"/"Enrich" pattern), and a name-based org-match step that runs ahead of the third-party lookups.

This spec draws a hard line between two clocks that must never be conflated:
- **The company-level cache clock** (`factual_fetched_at`/`dynamic_fetched_at` on `CompanySignals`) governs whether DASHR needs to hit the *public source* again at all. It is keyed to the shared cache row, never to a user's click â€” if User A's card enriches a company today, User B's unrelated card matching that same company next week sees it as already fresh and triggers no re-fetch and no charge. This is what keeps the third-party-lookup cost shared and low, per `CLAUDE.md`'s "cache once, reuse" rule.
- **The per-lead cooldown** (new `VisitingCard.company_enriched_at`, below) governs whether *this specific user's specific lead* is allowed to ask for its own company data (or score) to be re-pulled/recomputed again, independent of whether the shared cache actually needed a real re-fetch. It exists to stop a user re-triggering enrich/score on the same lead back-to-back, not to protect the third-party API â€” the action is billed every time it's used, whether or not it causes a real fetch underneath.

**Deliberately out of scope** (kept tight to the mechanisms `CLAUDE.md` names in this bullet plus this per-lead cooldown clarification, not a broader rework):
- No proactive/scheduled (Celery-beat) refresh job â€” the company-level refresh only happens when a card triggers the existing `POST /cards/{card_id}/enrich-company` CTA against an already-enriched, now-stale company. Nothing in `CLAUDE.md`'s wording mandates a cron; it only describes what a refresh does once triggered.
- No domain-based org matching. `SellerProfile` has no `domain`/`website` column today (confirmed by reading the model) â€” only `company_name` and `gst_no`. Matching in this spec is name-normalized-only; adding a domain to `SellerProfile` is a separate decision for a future spec.
- No change to `scoring.py`'s weighting logic. Preferring the linked org's declared data is scoped to the one `CompanySignals` column that already exists for it (`product_lines_summary`); teaching the scorer to weight a "this is a DASHR customer" signal is a distinct, separately-scoped change.
- No unlink flow. Once `linked_org_id` is set it is never cleared by this spec â€” `CLAUDE.md` describes it as a one-way tag, not a maintained relationship.
- No org-wide or global per-user cooldown counter â€” the cooldown is tracked **per lead** (`VisitingCard.company_enriched_at`), matching how the existing free correction-triggered rescore is already per-card, not per-user.

## API endpoints (apps/api)
- `POST /cards/{card_id}/enrich-company` â€” org-authenticated, **existing endpoint, behavior extended**. Today: 400 if the card has no `company_id`; 409 unless `company.enrichment_status == "pending"`. New behavior: also succeeds when `company.enrichment_status == "enriched"` **and at least one of these two independent reasons holds**:
  1. **Company-stale** â€” at least one of `factual_fetched_at`/`dynamic_fetched_at` is past its TTL. Only the stale tier(s) are re-fetched.
  2. **Lead-cooldown-elapsed** â€” this card's own `company_enriched_at` is non-null and at least `_LEAD_LEVEL_ACTION_COOLDOWN_DAYS` (30) days old. This can fire even when the company cache is still fully fresh, in which case `refresh_tiers` passed to the task is empty (nothing to actually re-fetch) â€” the call still runs, still succeeds, and is still billed exactly like any other enrichment action; it just re-confirms/re-attaches the current cache to this lead rather than calling any public source.

  Either reason **always bills** the triggering user via the existing `billing.charge_for_action(..., "enrichment", ...)` call â€” there is no free path here, and no branching in the billing call based on which of the two reasons fired or on whether a real fetch happened underneath. If `enrichment_status == "enriched"` and neither reason holds, the existing 409 (`CompanyNotEligibleForEnrichmentError`) is unchanged. `"enriching"`/`"failed"`/`"not_found"` statuses keep today's 409 behavior unchanged â€” this spec only adds these two extra paths for already-`"enriched"` companies, not a retry policy for the other terminal states. Response shape (`CardOut`) is unchanged; the frontend re-fetches `GET /cards/{card_id}` afterward, same as today.
- `POST /cards/{card_id}/score` â€” org-authenticated, **existing endpoint, behavior extended**. Today: first score is billed; if already scored, a rescore is allowed **and free** only when a `FieldCorrection` postdates `scored_at` (`CardAlreadyScoredError` otherwise). New behavior: when already scored, not correction-eligible, but this card's `company_enriched_at` is non-null and at least `_LEAD_LEVEL_ACTION_COOLDOWN_DAYS` (30) days old, a rescore is now allowed â€” but **billed** as a normal `"scoring"` action via `billing.charge_for_action`, unlike the free correction-triggered path. Both rescore paths remain mutually exclusive per call: a correction-eligible rescore stays free even if the cooldown has also elapsed (the free path is checked first, unchanged priority) â€” the billed cooldown path only fires when there is no pending free reason to rescore.
- `GET /cards/{card_id}` â€” org-authenticated, existing endpoint â€” `CardDetailOut` gains, at the top level alongside the existing `rescore_available` (unchanged, still correction-triggered/free):
  - `monthly_rescore_available: bool` (defaults `False`) â€” `true` when `lead_score is not None`, `rescore_available` is `false` (no free correction pending), and the lead-cooldown has elapsed. Only `get_card_detail` computes a real value, same as `rescore_available` today.

  `CardDetailOut.company` (`CardCompanyOut`) gains:
  - `is_linked_org: bool` â€” `true` when `Company.linked_org_id is not None`. The raw `linked_org_id` UUID is never returned â€” exposing it would let a seller viewing a prospect's card enumerate another org's internal id, which `CLAUDE.md`'s visibility rule for this linkage doesn't authorize; a boolean is the minimal useful signal for the review/export stage ("this prospect is already a DASHR customer").
  - `refresh_available: bool` (defaults `False`) â€” `true` when `enrichment_status == "enriched"` and **either** the company-stale reason **or** the lead-cooldown reason holds (both collapse into this one flag, since the frontend renders one button regardless of which reason unlocked it). Only `get_card_detail`'s single-card path computes a real value; `list_cards`' joined query doesn't (same rationale as `rescore_available` â€” avoiding a per-row fan-out query on every page load).

If no card-level behavior needs a new route, state so: no new endpoints beyond the two above are added; both are extensions of existing routes.

## Database changes
New migration `0024_company_linked_org_and_tiered_signals.py` (`down_revision = "0023"`):
- `companies`: add `linked_org_id: UUID | None`, `ForeignKey("organizations.org_id", ondelete="SET NULL")`, nullable, with index `ix_companies_linked_org_id`. No `org_id` is added to `companies` itself â€” `linked_org_id` is a one-way tag on the shared row, not a tenant scope (per `CLAUDE.md`'s explicit warning against ever adding `org_id` to `Company`).
- `company_signals`:
  - Add `factual_fetched_at: TIMESTAMPTZ | None`.
  - Add `dynamic_fetched_at: TIMESTAMPTZ | None`.
  - Data backfill: `UPDATE company_signals SET factual_fetched_at = updated_at, dynamic_fetched_at = updated_at WHERE updated_at IS NOT NULL` â€” every row enriched under the old one-shot model had both halves fetched together, so both new clocks start from that same timestamp.
  - Drop `updated_at` â€” fully superseded by the two tiered clocks; grepped and confirmed it has no other reader (`enrichment_service.py` is its only user, updated by this spec; no frontend or other service reads it).
- No `org_id` added to `company_signals` (unchanged â€” still the shared, non-tenant-scoped cache per spec 19's established convention).
- `visiting_cards`: add `company_enriched_at: TIMESTAMPTZ | None` â€” the per-lead cooldown anchor described in Overview. Deliberately **not** on `companies`/`company_signals`: it answers "since when has *this lead* had company data available to it," which is a per-card fact (a card created before the company was ever enriched, and one created after by riding on the same already-settled company, can legitimately have different values), not a property of the shared cache row.

## Background jobs
- `enrich_company_task` (`apps/api/app/workers/enrichment_processing.py`) â€” unchanged signature and trigger (still solely enqueued by `card_service.enrich_company_now`, never automatic). Internal behavior changes: its `enrichment_status != "pending"` guard for a *fresh* delivery (`self.request.retries == 0`) is loosened to also proceed when status is `"enriched"` and the caller determined (in `card_service.enrich_company_now`, before enqueueing) that a refresh is due â€” passed through as a new `refresh_tiers: list[str] | None` task argument (e.g. `["factual"]`, `["dynamic"]`, or `["factual", "dynamic"]`; `None` means "first-ever run, do both unconditionally", the existing behavior). The task does not itself flip `enrichment_status` to `"enriching"` for a refresh run â€” that transition stays exclusive to the true first-run to avoid it being surfaced to `"enriching"` for what is, from the frontend's perspective, a quick background top-up of an already-populated card.
- `enrichment_service.run_all_signal_lookups(db, company, gst_number, email_domain, website, products_offered, address, refresh_tiers=None)` â€” internal restructuring, no change to its two-value return (`signals, any_signal_found`):
  - The 14 existing lookups are grouped into two internal helpers by the bucketing `CLAUDE.md` specifies: `_run_factual_lookups` (registry, GSTIN, Udyam, website-derived product lines/plant size â€” lookups #1/#2/#3/#5, plus the `estimated_revenue_band` derivation which depends only on factual inputs) and `_run_dynamic_lookups` (LinkedIn, hiring/Naukri, GeM, Volza, Google News, Google Maps, the whole IndiaMART block, news summary, share price â€” lookups #4/#6â€“#14, plus `hiring_signal`/`marketplace_vintage_years`/`news_distress_detected`, all derived only from dynamic inputs).
  - `refresh_tiers=None` (first-ever run) calls both helpers, same total behavior as today, and sets both `factual_fetched_at`/`dynamic_fetched_at` to `now()`.
  - `refresh_tiers=["factual"]`/`["dynamic"]`/both calls only the requested helper(s) and only stamps the corresponding `*_fetched_at` column(s) â€” the other tier's data and clock are left untouched.
  - **Org-match step, run first, every call regardless of `refresh_tiers`** (cheap â€” DB-only, no external lookup cost): if `company.linked_org_id` is still `None`, calls new `match_linked_org(db, company)`; if it returns an `Organization`, sets `company.linked_org_id` on the spot. This runs "before falling back to third-party firmographics providers" per `CLAUDE.md` â€” concretely, it runs before `_run_factual_lookups` decides whether to call `website_signal_provider` for `product_lines_summary`.
  - `match_linked_org(db, company) -> Organization | None` â€” new function. Reuses `extraction_service._normalize_company_name`'s whitespace/case normalization (imported, not duplicated) to compare `company.normalized_name` against each candidate `SellerProfile.company_name`. Query: join `SellerProfile` â†’ `User` â†’ `Organization`, filter to users with a non-null `org_id`, compare normalized names, return the matched `Organization`. When a `Company` matches more than one org's declared name (rare â€” two customers claiming the same normalized company name), the match is skipped entirely (returns `None`) rather than picking one arbitrarily â€” an ambiguous/incorrect link would be worse than staying unlinked, and re-checked on the next enrichment run in case one side's name changes.
  - When `_run_factual_lookups` runs (first-run, or a factual-tier refresh) and `company.linked_org_id` is set: skip the `website_signal_provider` call for `product_lines_summary` entirely, and instead set `signals.product_lines_summary` directly from that org's declared `SellerProfile.product_lines` â€” preferring the org's own admin's profile (`User.role == "admin"`) if populated, else any member's populated profile, else falls through to the normal website-scrape behavior if no member has filled in `product_lines` yet (never leave the field silently null when the scrape would have found something). `plant_size_signal` is untouched â€” `SellerProfile` has no equivalent field to prefer.
- **Trigger**: still explicit-only. `card_service.enrich_company_now` computes whether this call is a first-run or a refresh (and, if a refresh, which tiers are actually stale â€” independent of *why* the call was allowed through) by reading the existing `CompanySignals` row before deciding to charge/enqueue, then passes `refresh_tiers` through to `enrich_company_task.delay(...)`. `refresh_tiers` can legitimately be `[]` (lead-cooldown-only trigger, company cache still fully fresh) â€” the task still runs its bookkeeping (stamping `company_enriched_at`, see below) even when it calls neither lookup helper.

**`company_enriched_at` stamping** (the per-lead cooldown anchor), in three places:
- `extraction_service._apply_new_lead` (the `card.company_id = company.company_id` assignment, `extraction_service.py:428`) and `_merge_fill_gaps` (the equivalent `canonical.company_id = company.company_id` assignment, `extraction_service.py:331`) â€” immediately after either assignment, if `company.enrichment_status in ("enriched", "not_found")`, stamp that card's `company_enriched_at = datetime.now(timezone.utc)`. Covers a card whose matched company was already settled by the time this card was parsed/merged â€” it never itself triggers `enrich_company_task`, so this is its only path to getting a cooldown anchor at all. Both call sites resolve the same company via the same `_get_or_create_company`, so this is one small helper (e.g. `_stamp_company_enriched_at(card, company)`) called from both, not duplicated logic.
- `enrich_company_task`'s success path (after `company.enrichment_status` is set to a terminal state, `enrichment_processing.py:180`) â€” two updates, in this order:
  1. Bulk-update: every `VisitingCard` row with `company_id == company.company_id` **and `company_enriched_at IS NULL`** gets `company_enriched_at = now()`. Covers every sibling card riding on this company (including ones created before this run) that hasn't yet had a cooldown anchor set â€” a single `UPDATE ... WHERE company_id = :cid AND company_enriched_at IS NULL`, not a per-row Python loop.
  2. The triggering card itself (`source_card_id`, if present) is then set to `company_enriched_at = now()` **unconditionally** (overwriting, not just filling null) â€” this is what actually restarts its 30-day cooldown on a lead-cooldown-triggered refresh; step 1 alone would've left it unchanged since it already has a non-null value from its own first run.

## Rules for implementation
- Every query on `Company`/`CompanySignals` still filters by `company_id` only, never `org_id` â€” `linked_org_id` is a tag, not a tenant scope; do not add `org_id` to either table.
- `match_linked_org`'s join through `SellerProfile`/`User`/`Organization` reads that data for enrichment purposes only â€” it must never be reused to expose one org's `SellerProfile`/leads/wallet/invoices to another org, and no new endpoint in this spec returns anything from the matched org beyond the single `product_lines` string folded into the shared `CompanySignals.product_lines_summary` cache and the `is_linked_org` boolean.
- The two TTLs are named constants in `enrichment_service.py` (`_FACTUAL_SIGNALS_TTL_DAYS = 180`, `_DYNAMIC_SIGNALS_TTL_DAYS = 90`), not a new configurable-data table â€” unlike scoring weights/pricing rates, `CLAUDE.md` states these as fixed figures, not something orgs will tune. The per-lead cooldown is a separate named constant, `_LEAD_LEVEL_ACTION_COOLDOWN_DAYS = 30`, in `card_service.py` (where both `enrich_company_now` and `score_card_now` read it) â€” not in `enrichment_service.py`, since it's a lead/billing-eligibility concept, not a signal-freshness one.
- **Never conflate the two clocks.** The company-stale check (`factual_fetched_at`/`dynamic_fetched_at` vs their TTLs) and the lead-cooldown check (`VisitingCard.company_enriched_at` vs `_LEAD_LEVEL_ACTION_COOLDOWN_DAYS`) are computed independently and only combined with an `or` at the eligibility-gate level in `enrich_company_now`. `run_all_signal_lookups`/`refresh_tiers` only ever look at the company-stale check â€” the lead-cooldown reason can unlock the *call*, but it never by itself forces a tier into `refresh_tiers`; a lead-cooldown-only trigger passes `refresh_tiers=[]`.
- A refresh/re-enrich is billed exactly like any other enrichment action via the existing `billing.charge_for_action(..., "enrichment", ...)` call in `enrich_company_now`, and a lead-cooldown rescore is billed via the existing `billing.charge_for_action(..., "scoring", ...)` call in `score_card_now` â€” in both cases every trigger reason bills the same way, with **no branching on whether a real fetch/recompute happened underneath or the data just came from cache** (explicit product decision â€” the charge reflects the user's action, not the marginal work done). This is unrelated to, and does not change, the existing free correction-triggered rescore or the free correction-triggered IndiaMART re-fetch â€” those stay exactly as `20-field-correction` defined them.
- The lead-cooldown check is evaluated **after** the free correction-triggered rescore check in `score_card_now` (unchanged priority) â€” a card that qualifies for both gets the free path, never the billed one.
- `company_enriched_at`'s bulk "fill nulls for every sibling card" update and the triggering card's own unconditional overwrite (see Background jobs) must stay in that order and both execute inside `enrich_company_task`'s existing single DB session/commit â€” never split across two separate commits that could leave siblings stamped but the triggering card not, or vice versa.
- No raw SQL string interpolation â€” SQLAlchemy query builder or bound params only.
- Business logic (staleness computation, tier selection, org matching) lives in `enrichment_service.py`/`card_service.py`, never in `routers/cards.py` or the Celery task body beyond the existing thin status-transition role `enrichment_processing.py` already has.
- `enrich_company_task` remains the only caller of `run_all_signal_lookups`'s refreshed/full fan-out â€” no other code path re-fetches signals outside this task and the existing, untouched `rerun_indiamart_supplier_profile_task`.
- API contract changes are the `CardCompanyOut` Pydantic model â€” the matching TS type in `lib/api.ts` is hand-aligned to match, never assumed.

- [ ] A brand-new company (`enrichment_status="pending"`) enriched via `POST /cards/{card_id}/enrich-company` ends with both `factual_fetched_at` and `dynamic_fetched_at` set to (approximately) the same timestamp, same as today's single-clock behavior
- [ ] Calling `POST /cards/{card_id}/enrich-company` again immediately after (same company, still fresh) returns `409` â€” unchanged one-shot behavior when nothing is stale
- [ ] Manually back-dating a company's `factual_fetched_at` to >180 days ago (dynamic left fresh) and re-triggering enrichment via a new card matched to that company re-runs only the factual lookups (verified by only `registry`/`gstin`/`udyam`/`website` `company_enrichment` audit rows being written on this run, and only `factual_fetched_at` advancing) â€” `dynamic_fetched_at` and all dynamic-tier `company_signals` columns are untouched
- [ ] Same test with `dynamic_fetched_at` back-dated instead: only dynamic-tier lookups re-run, `factual_fetched_at` untouched
- [ ] `GET /cards/{card_id}` for that company returns `refresh_available: true` while a tier is stale, and `false` once both clocks are fresh again
- [ ] Creating a `SellerProfile` (via an org admin) whose `company_name` normalizes to match an existing, previously-unlinked `Company`, then re-triggering that company's enrichment, sets `Company.linked_org_id` to that org and populates `CompanySignals.product_lines_summary` from `SellerProfile.product_lines` instead of the website scrape (verified by no `website` source `company_enrichment` audit row being written on that run)
- [ ] `GET /cards/{card_id}` for a card linked to that company returns `is_linked_org: true` and never returns the raw `linked_org_id`/org id anywhere in the response
- [ ] Two different orgs' `SellerProfile.company_name` normalizing to the same value as one `Company` results in `linked_org_id` staying `None` (no ambiguous link made) â€” verified by a unit test on `match_linked_org` directly
- [ ] A refresh run (`refresh_tiers` not `None`) debits the triggering user's wallet/free-allowance via the normal `"enrichment"` action path, verified by a new `WalletTransaction`/incremented `FreeActionAllowance.used_count` exactly as a first-run enrichment would produce
- [ ] `apps/web` card detail drawer shows the "Registered DASHR customer" badge for a linked company and the "Refresh Company Data" button only when `refresh_available` is true â€” verified by opening a card in the browser in both states
