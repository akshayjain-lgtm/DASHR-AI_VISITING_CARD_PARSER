# Spec: Parsing Visiting Card

## Overview
Implements the second workflow step in the DASHR AI pipeline (capture ‚Üí **extraction** ‚Üí enrichment ‚Üí scoring ‚Üí review/export). The `process_card` Celery task stood up as a no-op placeholder in `04-visiting-card-bulk-upload` becomes real: it loads a card's stored image, sends it to a vision-capable Claude model, and runs the model's JSON output through a deterministic validator/normalizer pass before persisting anything. Successful cards get `full_name`/`job_title`/`designation_level`/`website`/`address`/`products_offered`/`special_remark` (handwritten notes) filled in, their emails/phones written to `card_emails`/`card_phones`, and get linked to a `companies` shell row (name only ‚Äî firmographics stay `enrichment_status='pending'` for `06-company-enrichment` to fill in). Cards the model can't read, or that don't look like a business card at all, are marked `failed` with a stored reason rather than silently producing empty/garbage rows. Two special cases are handled so a physical batch of scans doesn't turn into inflated lead counts: a photo that is the **back side** of a card already photographed in the same batch gets its fields folded into that front card's row instead of becoming its own lead, and a card that's a genuine **re-scan of a contact already captured** gets folded into the existing lead the same way. This step does **not** call any enrichment provider and does **not** compute a lead score ‚Äî both remain untouched (`company_id` stays a bare shell row, `lead_score`/`score_breakdown`/`scored_at` stay `NULL`).

## Depends on
- `04-visiting-card-bulk-upload` ‚Äî needs the `visiting_cards` row + stored image this step's task body now actually acts on. Reuses `storage_service`, `scope_to_visible_users`, and the `ExhibitionNotFoundError`-style exception-per-failure-mode pattern from `services/exceptions.py`. **Amendment**: `04`'s upload endpoint no longer calls `process_card.delay(card_id)` itself ‚Äî this step adds the explicit `POST /cards/process` endpoint below as the only place (besides `POST /cards/{card_id}/reprocess`) that enqueues the task, so a batch sits at `status='new'` until the caller deliberately starts extraction.
- `01-database-setup` ‚Äî the `visiting_cards` columns this step populates (`full_name`, `job_title`, `designation_level`, `raw_ocr_text`, `special_remark`, `company_id`) and the `card_emails`/`card_phones`/`companies` tables already exist in the committed schema. `special_remark` was already documented there as "handwritten/marginal note captured off the card" ‚Äî this step is simply what populates it; it is **not** a new field and is not renamed to `handwritten_remark` (that's the concept, not a new column name).

## API endpoints (apps/api)
- `POST /cards/process` ‚Äî org-authenticated ‚Äî the explicit "start parsing" action; body `{ exhibition_id? }`. Enqueues `process_card` for every `status='new'` card visible to the caller (own cards, or every org member's if the caller is an admin ‚Äî same `scope_to_visible_users` rule as `GET /cards`), narrowed to one exhibition if `exhibition_id` is given, otherwise across all of the caller's visible cards regardless of exhibition (same "omitted filter = no scoping" convention `GET /cards?exhibition_id=` already uses). A card's enqueue failure (broker outage) is logged and skipped rather than failing the whole call, same as `POST /cards/{card_id}/reprocess`. Returns `200` with `{ enqueued_count }` ‚Äî the number of `'new'` cards matched and attempted, regardless of whether each individual `.delay()` call actually reached the broker. This is the **only** trigger for a freshly-uploaded batch's extraction ‚Äî `POST /cards/bulk-upload` (`04-visiting-card-bulk-upload`) never calls it itself.
- `GET /cards/{card_id}` ‚Äî org-authenticated ‚Äî full extraction detail for one card: all `CardOut` fields plus `designation_level`, `special_remark`, `website`, `address`, `products_offered`, `raw_ocr_text`, `extraction_error`, `merged_into_card_id` (set only for `status in ('merged', 'duplicate')`), `company` (`{ company_id, name, domain, website, enrichment_status }` or `null`), `emails` (`[{ email, email_type, is_primary }]`), `phones` (`[{ phone_e164, phone_raw, phone_type, is_primary }]`). Same visibility rule as `GET /cards` (own cards, or every org member's cards if caller is an admin); `404` if the card doesn't exist or isn't visible to the caller. Works for `merged`/`duplicate` cards too (returns their own ‚Äî now sparse ‚Äî detail plus where they were folded into), so a user can trace what happened to a scan that disappeared from the main list.
- `GET /cards` ‚Äî **behavior change**: when the `status` query param is omitted, results now exclude `status in ('merged', 'duplicate')` by default, so back-of-card scans and re-scans of an already-captured contact don't inflate the lead list. Passing `status=merged` or `status=duplicate` explicitly still returns them (for audit/debugging).
- `POST /cards/{card_id}/reprocess` ‚Äî org-authenticated ‚Äî re-enqueues extraction for a card whose `status == 'failed'`. Resets `status` to `'new'`, clears `extraction_error`, and calls `process_card.delay(card_id)` again. `404` if not found/not visible; `409` if the card's current status isn't `'failed'` (can't reprocess a card that's still `'new'`/`'processing'`, already succeeded, or was folded into another card as `'merged'`/`'duplicate'` ‚Äî reprocessing a folded card would be meaningless since its data already lives on the canonical row).

## Frontend surface (apps/web)
- **New**: `components/card-detail-drawer.tsx` ‚Äî slide-over panel opened by clicking a row in the upload page's card list. Shows every field from `GET /cards/{card_id}`: name/title/designation badge, company name + enrichment status pill, website/address/products-offered, handwritten remark, emails/phones list, and (for `status === 'failed'`) the `extraction_error` message plus a "Retry" button wired to `POST /cards/{card_id}/reprocess`.
- **Modified**: `app/upload/page.tsx` ‚Äî rows in the card list become clickable (opens `CardDetailDrawer` for that `card_id`). While any listed card has `status` in `('new', 'processing')`, poll `listCards()` every 4s (clear the interval once none remain) so extraction results appear without a manual refresh. Failed cards show a red status pill instead of the current plain uppercase text. Since `GET /cards` now excludes `merged`/`duplicate` rows by default, the list naturally shows one row per real contact ‚Äî no client-side dedup needed. A **"Parse Cards" button** renders above the card list whenever any listed card has `status='new'`; clicking it calls `POST /cards/process` (scoped to `selectedExhibitionId` if one is selected, matching the same filter already applied to the visible list) and then refreshes the list. The post-upload success banner no longer implies extraction started automatically ‚Äî it prompts the user to click "Parse Cards" next. Parsing is never triggered as a side effect of the upload call itself.
- **Modified**: `lib/api.ts` ‚Äî add `getCard(cardId)`, `reprocessCard(cardId)`, and `processCards(exhibitionId?)`, plus a `CardDetailOut` type (including `website`, `address`, `products_offered`, `merged_into_card_id`) and `CardOut.status` documented to include `"processing" | "extracted" | "failed" | "duplicate" | "merged"` alongside the existing `"new"`.

## Database changes
All additive, on top of the existing (already-implemented) schema ‚Äî no `org_id` columns, following `01-database-setup`'s documented user/admin-join visibility model (same deviation `04-visiting-card-bulk-upload` already followed).

- `visiting_cards.website` ‚Äî `TEXT`, nullable. Company/personal website as printed on the card.
- `visiting_cards.address` ‚Äî `TEXT`, nullable. Full postal address as printed on the card. Kept per-card rather than on `companies` because different reps/branches of the same company legitimately print different office addresses ‚Äî the shared `companies` cache stays limited to `hq_city`/`hq_country` (set later by enrichment), never overwritten by one card's specific branch address.
- `visiting_cards.products_offered` ‚Äî `TEXT`, nullable. Free text for "what the company deals in" when printed/stated on the card (e.g. "Manufacturers of industrial valves & fittings"). Kept per-card, not on `companies`, for the same reason as `address` ‚Äî it's promotional text from one specific card, not a verified firmographic; `06-company-enrichment` owns anything that should live on the shared row.
- `visiting_cards.upload_batch_id` ‚Äî `UUID`, nullable. One value generated per `bulk_upload_cards()` call, shared by every card created from that request. Exists purely to let extraction correlate photos uploaded together (see back-of-card handling below) ‚Äî not a foreign key, just a correlation id.
- `visiting_cards.batch_sequence` ‚Äî `INTEGER`, nullable. This card's 0-indexed position within `files` in its `bulk_upload_cards()` call. Used to find the "previous photo in the batch" when checking whether a card is the back side of the one just uploaded before it.
- `visiting_cards.merged_into_card_id` ‚Äî `UUID`, nullable, `FOREIGN KEY ‚Üí visiting_cards.card_id`. Set on a card whose data was folded into another card's row (either a back-side match or a duplicate match) instead of being kept as its own lead. Points at the canonical row. `NULL` for every card that stands on its own.

New index: `ix_visiting_cards_upload_batch_id` on `visiting_cards (upload_batch_id)` ‚Äî the lookup this step's back-of-card matching runs per card.

New Alembic revision: `0006_card_extraction_fields.py`, `down_revision = "0005"`.

No new columns on `companies` ‚Äî extraction only ever writes `name`/`normalized_name`/`website` (backfilled only when currently `NULL`) on get-or-create; `enrichment_status` already defaults to `'pending'` from `01-database-setup`.

## Background jobs
- `apps/api/app/workers/card_processing.py::process_card(card_id)` ‚Äî no longer a placeholder. Loads the card; if `status` isn't `'new'`, returns immediately (idempotency guard against duplicate Celery delivery). Sets `status='processing'`, commits, then calls `extraction_service.extract_card(db, card)`, which:
  1. Downloads + downscales the image, calls the vision model, validates/normalizes every field (see Files to create for the exact shape).
  2. If the model's output looks like a **back-of-card** (no name/title/emails/phones extracted, or the model's own `is_back_of_card` flag is true) ‚Äî looks for a sibling card with the same `upload_batch_id` and `batch_sequence == this_card.batch_sequence - 1` that isn't itself `failed`/`merged`/`duplicate`. If found: merges this card's extracted fields onto the sibling (front card wins on any field both have; this card only fills gaps ‚Äî including adding any emails/phones/website/address/products_offered/special_remark the front didn't already have), sets *this* card's `status='merged'`, `merged_into_card_id=<front card id>`, `processed_at=now()`, and does not create a `companies` row or `card_emails`/`card_phones` rows under this card's own id. If no such sibling exists (e.g. this was the first photo in the batch, or the true front was uploaded in a different batch), it's processed as an ordinary card instead ‚Äî better to keep a lead with only address/website/products than silently drop it.
  3. Otherwise (looks like an ordinary front-of-card) ‚Äî checks for a **duplicate**: an existing card visible to the same user (via `scope_to_visible_users`, so an admin's dedup check spans their whole org) with `status not in ('failed', 'merged', 'duplicate')` matching on, in priority order: same primary email (case-insensitive) ‚Üí same primary `phone_e164` ‚Üí same `(normalized full_name, company_id)`. If a match is found: same fill-gaps-only merge onto the existing card, this card gets `status='duplicate'`, `merged_into_card_id=<existing card id>`, `processed_at=now()`, no new `companies`/`card_emails`/`card_phones` rows under this card's own id.
  4. Otherwise: proceeds as a normal new lead ‚Äî get-or-creates the `companies` row by `normalized_name` (backfilling `website` if the existing row's is `NULL`), writes `card_emails`/`card_phones`, sets `designation_level`, and marks `status='extracted'`, `processed_at=now()`.
- On a permanent failure (`ExtractionValidationError` ‚Äî unreadable/non-card image, or a well-formed-but-empty model response with no name, no company, no contact info, no address/website/products either), catches it, sets `status='failed'`, `extraction_error=str(exc)`, `processed_at=now()`, no retry. On a transient failure (`VisionApiError` ‚Äî timeout, rate limit, 5xx), lets Celery retry via `autoretry_for=(VisionApiError,)`, `retry_backoff=True`, `max_retries=3`; only after retries are exhausted does it fall through to the same `status='failed'` handling.
- Triggered exactly as before: once per card, immediately after `card_service.bulk_upload_cards` commits the row (no change to the enqueue site ‚Äî `upload_batch_id`/`batch_sequence` are stamped onto the row at creation time, not by the task), plus now also by the new `POST /cards/{card_id}/reprocess` endpoint (a reprocessed card keeps its original `upload_batch_id`/`batch_sequence`, so it can still be matched as a back side or duplicate on retry).

## Files to create
- `apps/api/migrations/versions/0006_card_extraction_fields.py`
- `apps/api/app/services/vision_client.py` ‚Äî thin wrapper around the Anthropic API: builds the vision message (base64 image + extraction prompt asking for a strict JSON shape ‚Äî name, title, company, website, address, products_offered, handwritten remark, emails[], phones[], and a boolean `is_back_of_card`), calls the model, returns the raw parsed JSON or raises `VisionApiError` for timeouts/rate-limits/5xx. No validation logic lives here ‚Äî just the API call and response-shape parsing.
- `apps/api/app/services/extraction_service.py` ‚Äî orchestrates one card's extraction: downloads the image via `storage_service`, downscales it with Pillow (already a dependency; the `pillow-heif` plugin registered via `pillow_heif.register_heif_opener()` at module import time lets Pillow open HEIC/HEIF source images from `04-visiting-card-bulk-upload`, since Pillow has no built-in HEIC decoder) to a max 1568px edge, and **always re-encodes to JPEG regardless of source format** before base64-encoding and calling `vision_client` ‚Äî this isn't just a size optimization for large phone photos; Anthropic's vision API doesn't accept HEIC/HEIF as an input media type at all, so this re-encode step is what makes a HEIC upload extractable in the first place, not merely cheaper. Then it runs the validator/normalizer pass (trims/blank-rejects name/title/company/website/address/products_offered; validates each email via `email-validator`, keeping only well-formed ones; normalizes each phone to E.164 via `phonenumbers`, keeping the raw string alongside when it can't be parsed; first surviving email/phone is `is_primary`), classifies seniority via `designation.py`, then runs the back-of-card check and the duplicate check described in Background jobs before deciding whether to create a new `companies`/`card_emails`/`card_phones` set or fold this card's fields into an existing row. Raises `ExtractionValidationError` if the model returns no name, no company, no contact info, and no address/website/products at all ‚Äî signals "this wasn't a business card," not a bug to swallow.
- `apps/api/app/services/designation.py` ‚Äî a static keyword‚Üíseniority-level table (`{"c_level": [...], "director": [...], "manager": [...]}`, fallback `"individual_contributor"`) and a `classify(job_title: str | None) -> str | None` function. Kept as data here (same reasoning as CLAUDE.md's scoring-weights rule) so `05-lead-scoring`'s seniority criterion can reuse it without re-deriving levels from raw titles.
- `apps/web/components/card-detail-drawer.tsx`

## Rules for implementation
- Every query on `visiting_cards`/`card_emails`/`card_phones` continues to go through `scope_to_visible_users`'s admin-sees-org-member join ‚Äî including the new duplicate-detection lookup, which must never search outside the caller's own visibility scope (no cross-org dedup).
- No raw SQL string interpolation ‚Äî SQLAlchemy query builder or bound params only.
- All extraction/validation/normalization/merge logic lives in `services/` (`vision_client.py`, `extraction_service.py`, `designation.py`) ‚Äî `card_processing.py` stays a thin task that loads the card, calls the service, and maps outcomes to `status`/`processed_at`/`extraction_error`.
- Never trust raw vision-LLM output as final ‚Äî every field goes through the validator/normalizer pass in `extraction_service.py` before touching `visiting_cards`/`card_emails`/`card_phones`; a malformed email or unparseable phone is dropped/flagged, never persisted as-is.
- The upload request path is untouched ‚Äî extraction, back-of-card matching, and duplicate detection only ever run inside the Celery task, never inline in a request handler.
- A merged or duplicate card's own row is never deleted ‚Äî it stays for image/audit trail with `status`/`merged_into_card_id` set, and is simply excluded from the default `GET /cards` listing. Merging is additive/fill-gaps-only onto the canonical row; it never overwrites a field the canonical row already has.
- `companies` get-or-create matches on `normalized_name` only in this step (no domain-based fuzzy matching, no firmographics lookup) ‚Äî that logic is `06-company-enrichment`'s job, not this one's. The one exception is opportunistically backfilling `companies.website` when the existing row's is `NULL`, since that's the same printed fact, not an inference.
- API contracts are Pydantic models (`CardDetailOut` etc.) ‚Äî the TS types in `lib/api.ts` are hand-aligned to match, never assumed.
