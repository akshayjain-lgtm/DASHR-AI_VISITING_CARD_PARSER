# Spec: Data Enrichment

## Overview
Fills in the third stage of the roadmap (capture → extraction → **enrichment** → scoring → review/export): once a card is extracted and linked to a `companies` row, this step pulls that company's public footprint from a fixed set of free, publicly accessible Indian sources — government registries/portals and public-facing web pages, **no paid or commercial data API** — and persists each finding as its own typed signal so a later scoring step can weight them individually and explain a score in plain terms. The `companies` and `company_enrichment` tables, and the `enrichment_status`/`enriched_at` columns, already exist (`01-database-setup`); `05-parsing-visiting-card`'s `extraction_service._get_or_create_company` already creates a bare `Company` row (`name`, `normalized_name`, `website`) with `enrichment_status='pending'` whenever a card names a company DASHR hasn't seen before. This step resolves that `'pending'` state: a background Celery task fans out across nine provider modules (eleven independently-guarded lookups, since two modules each expose two distinct lookups), stores each one's raw response for audit, flattens the results onto a new `company_signals` table (one column per signal, ready for scoring to read directly), and generates a short human-readable summary a seller can read at a glance.

**No paid API integration is planned for any of these sources** — this is a hard constraint from the business, not a temporary stub-until-funded state. Every source here is either a government portal that publishes free public search (MCA, GST, Udyam, GeM) or a public web page/directory that doesn't require login or payment to view (Zauba Corp, LinkedIn company/job pages, Naukri job listings, Google News, Google Maps search results, IndiaMART/TradeIndia/JustDial). Three sources from an earlier draft of this spec are **dropped** because they have no meaningful free tier and would require a paid subscription to use at all: Probe42, Tofler, and CorpVeda (commercial financial-data aggregators). Revenue signal is instead derived, for free, from Udyam's public investment/turnover-based classification (`udyam_category`) — a coarser signal than an exact revenue estimate, but a real public one. Similarly, Volza/ImportGenius trade-data is scoped to only the free teaser numbers their public company-profile pages show (e.g. "X shipments in the last 12 months") — the detailed shipment-level records both platforms sell are out of scope.

Because every source here is a scrape of a public page or portal rather than a documented API, this step carries real engineering and legal caveats that a paid-API integration wouldn't: page structures change without notice and will break parsers over time; several sites (LinkedIn, Google Maps, and parts of Naukri) render results client-side in JavaScript and cannot be read with a plain HTTP GET; the GST and MCA public search portals are captcha-gated in practice, which blocks straightforward automation; and scraping a site's public pages may run against that site's own Terms of Service even when no login/payment is involved — a legal/compliance question this spec flags but does not resolve. Every source therefore lands behind a `Protocol` with a stub that returns "no signal found," so the pipeline (task fan-out, audit logging, `company_signals` population, summary generation) is real and testable end-to-end from day one, while each real scraper is implemented and hardened per source — some (Udyam, GeM, Google News' public RSS, IndiaMART/TradeIndia/JustDial) are realistically buildable now; others (GST/MCA captcha, LinkedIn/Google Maps JS rendering) may need to ship as a stub for longer while that per-source friction is resolved.

## Depends on
- `01-database-setup` — `companies` (`company_id`, `name`, `normalized_name`, `domain`, `website`, `industry`, `size_bucket`, `hq_city`, `hq_country`, `linkedin_url`, `enrichment_status`, `enriched_at`) and `company_enrichment` (`enrichment_id`, `company_id`, `source`, `payload` JSONB, `fetched_at`) already exist at migration head `0007`.
- `05-parsing-visiting-card` — `extraction_service._get_or_create_company` already creates/looks up `Company` rows and sets `visiting_cards.company_id`; the card's already-captured `gst_number` (`06`/card extraction fields) is the input to GSTIN verification below. This step reacts to `Company.enrichment_status`, it doesn't change how cards get linked to a company.

## API endpoints (apps/api)
- `POST /cards/{card_id}/enrich-company` — org-authenticated (existing `get_visible_card` scoping, unchanged) — the explicit "Enrich Company" CTA. Enrichment is **never** triggered automatically after parsing — a seller has to ask for it per card, mirroring how "Parse Cards" is itself a separate explicit action from upload rather than an invisible side effect of the previous stage. Requires `card.company_id` to be set (`400` if not — nothing to enrich yet) and the linked `Company.enrichment_status == "pending"` (`409` if it's already `"enriching"`/`"enriched"`/`"not_found"`/`"failed"` — this is a one-shot action per company, not a re-run button). Enqueues `enrich_company_task.delay(company_id, card_id)` and returns the unchanged `CardOut` — the frontend re-fetches `GET /cards/{card_id}` afterward to pick up the new status.
- `GET /cards/{card_id}` — org-authenticated (existing auth, unchanged) — `CardDetailOut.company` (`CardCompanyOut`) gains:
  - `summary: str | None` — the generated human-readable digest (see Background jobs), the primary thing a seller reads
  - `summary_generated_at: datetime | None`
  - A small set of headline signals for at-a-glance badges, pulled from `company_signals`: `linkedin_employee_count: int | None`, `estimated_revenue_band: str | None`, `gstin_verified: bool | None`, `udyam_registered: bool | None`, `hiring_signal: str | None`, `google_rating: float | None`
  - `enrichment_status` gains two new possible values (`"enriching"`, `"not_found"`) alongside the existing default `"pending"` and eventual `"enriched"`/`"failed"`
- The full `company_signals` row (all raw columns) is **not** pushed to this endpoint — it's scoring's internal input, read straight from the DB by the future scoring step, not a UI surface. If a later step needs to show every raw signal to a seller, that's a new endpoint for that step to add, not this one.

## Frontend surface (apps/web)
- **Modified**: `components/card-detail-drawer.tsx` — the existing "Company" section (currently just `name` + `enrichment_status`) renders, based on `enrichment_status`: `"pending"` shows an explicit **"Enrich Company" button** (the CTA, calling the new endpoint then re-fetching the card); `"enriching"` shows an "Enriching…" hint (no button — already in flight); `"not_found"` shows "No public data found"; `"failed"` shows a plain "Enrichment failed" note (no retry button — re-running a failed enrichment is out of this step's scope, matching Background jobs); `"enriched"` renders the `summary` paragraph plus a compact badge row for the headline signals (employee count, revenue band, a "GSTIN verified" / "Udyam registered" checkmark pair, hiring signal, Google rating) when each is non-null.
- **Modified**: `lib/api.ts` — `CardCompanyOut` type gains the same new fields to match the backend schema; adds `enrichCompany(cardId)` calling the new endpoint.

No new pages.

## Database changes
- New migration `0008`:
  - Adds `summary: str | None` and `summary_generated_at: datetime | None` to `companies` — the one generated-text field per company, kept with the identity row since it's 1:1 and small.
  - Creates `company_signals` (new table, **no `org_id`** — see below): one row per company (`company_id` unique FK to `companies.company_id`, `ondelete="CASCADE"`), flattened, individually-typed columns so scoring never has to parse a source's raw HTML/JSON shape:
    - *Registry (MCA public master-data search, Zauba Corp public mirror as the primary practical source)*: `cin`, `incorporation_date`, `registry_status`, `registered_address`, `authorized_capital` (Numeric), `paid_up_capital` (Numeric)
    - *Compliance (GST portal public taxpayer search / Udyam public registration search)*: `gstin_verified` (bool), `gstin_status`, `udyam_registered` (bool), `udyam_category` (`"micro"`/`"small"`/`"medium"`)
    - *Firmographics (LinkedIn public company page)*: `linkedin_employee_count` (int), `linkedin_follower_count` (int)
    - *Revenue signal (derived, not fetched)*: `estimated_revenue_band` (str) — bucketed in `enrichment_service.py` from `udyam_category` (Udyam's own micro/small/medium bands are turnover-threshold-based) and `paid_up_capital` when present; **no** raw revenue amount column, since no free public source publishes one
    - *Website-derived*: `product_lines_summary`, `plant_size_signal` (free text — e.g. "Website references a 50,000 sq ft manufacturing facility"; no source publishes a structured "plant size" field, so this is text extracted from the company's own site, not a number to compute on)
    - *Growth/momentum (Naukri + LinkedIn public job-search pages / GeM public tender search / Volza-ImportGenius public teaser numbers / Google News public RSS)*: `active_job_postings_count` (int), `hiring_signal` (`"expanding"`/`"stable"`/`"unknown"`, a simple threshold bucketed in `enrichment_service.py`, not a black box), `gem_tender_count` (int), `gem_total_tender_value` (Numeric), `import_export_activity` (bool), `shipment_count_last_12m` (int — from the free teaser number only, not itemized shipments), `recent_news_signals` (JSONB list of `{headline, url, published_at, signal_type}`, `signal_type` one of `"funding"`/`"expansion"`/`"new_facility"`/`"other"`)
    - *Local presence (Google Maps public search results / IndiaMART-TradeIndia-JustDial public directory listings)*: `google_rating` (Numeric), `google_review_count` (int), `marketplace_vintage_years` (int), `marketplace_verified_badge` (bool), `marketplace_located_in_industrial_area` (bool — derived by pattern-matching the directory listing's address for industrial-estate markers like "MIDC"/"GIDC"/"Industrial Area"/"SIDCO", not a field any directory exposes directly)
    - `updated_at` (timestamp, bumped on every re-run)
  - `company_enrichment` needs no schema change — its existing `source`/`payload`/`fetched_at` shape already fits "one row per source per attempt"; this step just writes many more rows per company (one per successful lookup: `"mca"` or `"zauba"`, `"gstin"`, `"udyam"`, `"linkedin"`, `"naukri"` and/or `"linkedin_jobs"`, `"gem"`, `"volza"`/`"importgenius"`, `"google_news"`, `"google_maps"`, `"indiamart"`/`"tradeindia"`/`"justdial"` (whichever answered)) instead of one row per company.
- `companies`/`company_enrichment`/`company_signals` intentionally carry **no** `org_id` — per `CLAUDE.md`'s data model essentials, `Company` is a shared cache keyed by normalized name/domain, reused across orgs where public data allows. This is a deliberate, documented exception to the "every table must carry `org_id`" rule, not an oversight, and applies to every new table/column in this step for the same reason.

## Background jobs
- New Celery task `app.workers.enrichment_processing.enrich_company_task(company_id: str)`, fanning out over eleven independent public-source lookups (across nine provider modules):
  1. Loads the `Company` row; if it no longer exists or `enrichment_status` is not `"pending"`, returns immediately (idempotent guard, mirrors `process_card`'s `status != "new"` check — the cache-check `CLAUDE.md` requires before any repeated outbound lookup, paid or not, since these sources still risk rate-limiting/IP-blocking on repeated automated hits).
  2. Sets `enrichment_status = "enriching"` and commits before calling any source, so a concurrent duplicate enqueue (two cards for the same new company processed back-to-back) skips instead of double-scraping every source.
  3. Calls `enrichment_service.run_all_signal_lookups(db, company)`, which invokes each of the eleven source-lookups **independently, each wrapped in its own `try/except`** — one source being down, blocked, rate-limited, or erroring must never block the others from populating their columns. Every lookup that succeeds writes one `CompanyEnrichment` audit row (raw scraped payload, tagged `source`) and updates its slice of `company_signals`; a failed lookup writes nothing and simply leaves those columns null for this run.
  4. Upserts the single `company_signals` row for this company from whatever was collected (create on first enrichment, update in place on any future re-run).
  5. Calls `enrichment_summary.generate_summary(company, signals)` — a text-only Claude call (same `anthropic` client `vision_client.py` already uses, no image) that turns whatever signals exist into a 2–4 sentence plain-English summary (e.g. *"ABC Industries (incorporated 2008, GSTIN verified, Udyam-registered small enterprise) operates from an MIDC industrial estate near Pune. LinkedIn shows ~120 employees and 3 open engineering roles. No recent expansion news found."*) — best-effort over partial data, never blocked on a source that returned nothing.
  6. Sets `companies.summary`/`summary_generated_at`, and `enrichment_status = "enriched"` if at least one source returned a signal, or `"not_found"` if every source came back empty; sets `enriched_at = now()`; commits.
  - A failure *outside* the per-source guards (e.g. a DB error mid-upsert) retries like `process_card` (exponential backoff, max 3 attempts), finalizing `enrichment_status = "failed"` if retries are exhausted. `"failed"` is not auto-retried by a later card reusing this company (only `"pending"` triggers a re-run) — a manual re-enrich path is out of this step's scope.
- **Trigger**: explicit only — `POST /cards/{card_id}/enrich-company` (`card_service.enrich_company_now`) is the sole caller of `enrich_company_task.delay(...)`. `process_card` does **not** auto-enqueue enrichment after parsing; this was an earlier design in this spec, deliberately reversed so every pipeline stage (upload → parse → enrich) stays a distinct, visible, user-initiated action rather than an invisible side effect of the one before it — the same reason bulk upload doesn't auto-enqueue parsing. `enrich_company_now` resolves the card via the existing org-scoped `get_visible_card`, requires `card.company_id` to be set and the linked `Company.enrichment_status == "pending"`, then calls `enrich_company_task.delay(str(card.company_id), str(card.card_id))` (card id, not the raw GSTIN — see the GSTIN note above).
- Enqueue failures are logged and swallowed (same `try/except Exception: logger.exception(...)` pattern already used for `process_card.delay(...)`/`reprocess_card`'s re-enqueue) — a failed enqueue must never fail the request, which already validated the card/company state.

## New dependencies
- Promoted `httpx` (pip) from the existing test-only dependency to a real one — used by the one real source implemented so far (Google News RSS) and the Wikipedia summary-context lookup below.
- `beautifulsoup4`/`playwright` are **not** added — deferred until a source that actually needs them (real JS-rendered or HTML-heavy scraping) gets implemented, per the original incremental-delivery plan.

### Real-source status (post-implementation, network-verified — not assumed)
Of the eleven lookups, only **one is a real (non-stub) implementation today**: Google News RSS (`news_signal_provider.RealNewsSignalProvider`), confirmed reachable and parsed successfully in this deployment's network. The rest remain stubbed for concrete, verified reasons (`curl`-tested, not guessed) rather than "not gotten to yet":
- **Zauba Corp**: HTTP 403 with `cf-mitigated: challenge` — an active Cloudflare bot challenge, not solvable with a plain HTTP client.
- **MCA**: flat HTTP 403 (WAF/IP block, no challenge page).
- **GeM**: connection refused outright — no HTTP response at all from this network.
- **Udyam**: the portal itself is reachable, but its public tool only *verifies a given Udyam/UAM number* — there is no company-name search endpoint, so this isn't implementable from a company name regardless of network access.
- **GSTIN verification**: still believed captcha-gated (per original Overview), not re-tested since GSTIN itself is already scoped as "only when the card captured one."
- Every provider factory (`get_*_provider()`) is environment-gated: `settings.environment == "test"` always returns the stub (so the test suite never makes a live network call), any other environment returns the real implementation where one exists. This is the standing pattern for every source revisited going forward, not just Google News.

### Summary-only supplementary source (not a scoring signal)
Per explicit follow-up direction: the generated `summary` text (never `company_signals`/scoring columns) may also draw on public sources outside the eleven trusted signal sources, as long as the source is cited inline in the summary text. Implemented: **Wikipedia** (`enrichment_summary._fetch_wikipedia_context`) — a two-step lookup (MediaWiki full-text search to resolve a company's legal/registered name to its Wikipedia article title, e.g. "InterGlobe Aviation Limited" → "IndiGo", then that article's summary extract), fed into the Claude prompt with an explicit instruction to write "(Source: Wikipedia)" after anything drawn from it, and included the same way in the deterministic non-LLM fallback text if the Claude call itself fails. This directly addresses well-known/publicly-traded companies (airlines, banks, large listed firms) that legitimately have zero matches across the eleven India-SME-oriented/currently-blocked signal sources but obviously have real public information available elsewhere — `enrichment_status` still correctly reflects `"not_found"` in that case (no *signal* source answered), while `summary` can still be genuinely useful and clearly labeled as citing non-signal background. Same environment gate as the real providers (stubbed to `None` in tests).

## Rules for implementation
- Every query on `Company`/`CompanyEnrichment`/`CompanySignals` filters by `company_id` only — there is no `org_id` on these tables by design (see Database changes); do not add one or attempt to scope these queries per-org.
- `enrich_company_task` never scrapes any source for a `Company` whose `enrichment_status` is anything other than `"pending"` — this is the cache-check `CLAUDE.md` requires before any repeated outbound lookup, and it gates the whole fan-out, not just one source. Treat these public sources with the same restraint the rule was written for commercial APIs: repeated automated hits risk getting DASHR's IP rate-limited or blocked, which is its own form of "billed" cost.
- Each of the eleven source-lookups in `enrichment_service.run_all_signal_lookups` is wrapped in its own `try/except` — a raised exception from one source is logged and skipped, never allowed to propagate and abort the other lookups or the summary step.
- No raw SQL string interpolation — SQLAlchemy query builder or bound params only.
- All scraping, audit-logging, and column-mapping logic lives in `enrichment_service.py` (plus the provider modules themselves) — never in the Celery task or a router. `enrichment_processing.py` stays a thin status-transition-and-call layer, the same relationship `card_processing.py` has to `extraction_service.py`.
- `hiring_signal`, `estimated_revenue_band`, and any other bucketed/derived field's thresholds live as named constants in `enrichment_service.py`, not inline magic numbers, so they're one place to tune later (mirrors `CLAUDE.md`'s "scoring weights live as data, not inline branches" rule applied to signal derivation).
- Enrichment is background-only — never called synchronously from a request handler, and no new synchronous endpoint triggers it.
- Every provider's stub implementation returns "no signal" (`None`, or an empty/false result per its return type) — it must never fabricate plausible-looking data, which would silently corrupt scoring once a scoring step reads `company_signals`.
- Respect each site's `robots.txt` and keep request rates conservative (one lookup per company per source, no polling/retrying a source that already responded) — this doesn't eliminate the ToS exposure noted in Overview, but it's the minimum good-citizen behavior expected of a scraper hitting sites that weren't built for programmatic access.
- API contract changes are the `CardCompanyOut` Pydantic model — the matching `CardCompanyOut` TS type in `lib/api.ts` is hand-aligned to match, never assumed.
- Scoring (a future step) reads `company_signals` columns directly and must never re-parse `company_enrichment.payload` at score time — that payload is an audit trail for debugging/reprocessing, not a scoring input; this is why `company_signals` exists as a flattened table at all.

## Definition of done
- [ ] A card whose extracted `company_name` doesn't match any existing `Company` creates a new row with `enrichment_status='pending'`, and it **stays** `"pending"` indefinitely until a seller explicitly calls `POST /cards/{card_id}/enrich-company` — parsing alone never starts enrichment
- [ ] `POST /cards/{card_id}/enrich-company` on a card whose company is `"pending"` returns `200` and, within one Celery cycle, that company's `enrichment_status` becomes `"enriched"` or `"not_found"` (never left at `"pending"`)
- [ ] `POST /cards/{card_id}/enrich-company` on a card with no linked company returns `400`; on a card whose company is already `"enriching"`/`"enriched"`/`"not_found"`/`"failed"` returns `409` and does **not** enqueue another lookup — verified by no new `company_enrichment` rows being written for that `company_id`
- [ ] A run against `Company` with every source stubbed (default dev config) still produces exactly one `company_signals` row (all columns null) and a non-null `companies.summary` (a graceful "no public data found for this company yet" style message), with `enrichment_status="not_found"` and no unhandled exception in the worker log
- [ ] Simulating one source raising an exception (e.g. monkeypatching `registry_provider` to raise) while the others succeed still results in `enrichment_status="enriched"`, a populated `company_signals` row for the sources that succeeded, and a `companies.summary` generated from that partial data — the one failure never aborts the run
- [ ] A successful run writes one `company_enrichment` audit row per source that returned data (not one row for the whole run), each with the correct `source` tag and a non-null `payload`
- [ ] `GET /cards/{card_id}` for a card whose company is `"enriched"` returns `summary` and the six headline fields in its `company` object
- [ ] `apps/web` card detail drawer shows an "Enrich Company" button for `"pending"`, "Enriching…" for `"enriching"`, "No public data found" for `"not_found"`, "Enrichment failed" for `"failed"`, and the summary paragraph + badge row once `"enriched"` — verified by opening a card in the browser at each state
- [ ] `POST /cards/{card_id}/enrich-company` for a card belonging to another org returns `404` (existing `get_visible_card` org-scoping applies to this new endpoint too)
- [ ] Deleting a `Company` row cascade-deletes its `company_signals` row (FK `ondelete="CASCADE"`), leaving no orphaned signal row
- [ ] No settings/config entry in this step requires an API key, subscription, or paid credential of any kind — grep for `_api_key`/`_secret` additions in `core/config.py` from this step's diff and confirm there are none
