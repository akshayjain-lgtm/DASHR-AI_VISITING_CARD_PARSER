# Spec: Field Correction

## Overview
AI extraction and enrichment are never perfect — a vision-LLM misread on a card, or an IndiaMART storefront resolved via Apify's search-scraper cascade to the wrong (or a stale) URL. This step sits in the roadmap's Review stage, right after Extraction/Enrichment/Scoring and alongside Export: it lets a seller correct a mis-extracted or mis-enriched field inline on a card, without waiting for a future AI/enrichment re-run. Every correction writes both the pre-correction (`original_value`) and post-correction (`corrected_value`) value to a new, append-only `FieldCorrection` audit table, so extraction/enrichment accuracy can be measured field-by-field later (a future accuracy-report feature, out of scope here — this step only creates the data for it). Eight fields are correctable: `full_name`, `job_title` (role), `address`, `products_offered`, `company_name`, `email`, `phone`, and a company's IndiaMART URL (`catalog_url`). Correcting the IndiaMART URL is the one field with a side effect beyond the audit write: it re-triggers a real Apify `supplierProfile` lookup against the corrected URL, refreshing that company's `indiamart_*` signal fields.

**Phase 2 (billing amendment, done):** neither correction-driven side effect is billed. Correcting `catalog_url` no longer charges an `"enrichment"` action for the re-fetch — it's fixing a mistake in an already-paid-for enrichment, not a new billable action, and this holds even at 0 wallet balance with the free allowance exhausted. Separately, once a card has been scored, correcting *any* of the eight fields unlocks exactly one free rescore (`10-lead-scoring`'s one-shot rule gets this one carve-out) — also never billed, never counted against the free allowance — so a seller who fixes a mistake DASHR's own AI made doesn't pay to see the corrected score.

**Phase 3 (bugfix, done):** a `job_title` correction alone didn't move `designation_score` on a subsequent rescore — reported live ("changed role from Proprietor to Sales, designation score didn't change"). Root cause: `scoring._designation_score` reads `VisitingCard.designation_level`, a field derived from `job_title` by `designation.classify()` **once, at extraction time** (`extraction_service.py`), not stored redundantly in lockstep with `job_title` afterward. The `job_title` correction branch updated `job_title` but never re-ran `classify()`, so `designation_level` (and therefore the score) kept reflecting the pre-correction title even after Phase 2's free rescore ran. Fixed: correcting `job_title` now also sets `card.designation_level = designation.classify(corrected_value)` in the same transaction — a re-derivation, not a second `FieldCorrection` row (mirrors how `company_name`'s re-match isn't separately audited either; the `job_title` row's own original/corrected values are the complete audit trail). Existing-data note: a card whose `job_title` was corrected *before* this fix shipped still carries a stale `designation_level` — only a *new* `job_title` correction (or a data backfill, not built) re-derives it; this phase does not retroactively fix already-corrected rows.

**Phase 4 (`/code-review-feature` action plan, done):** a security + quality review of Phases 1-3 surfaced one real gap and several worth-fixing quality items, all addressed:
- **Security — unbounded, never-billed Apify spend.** Phase 2 removed the only throttle `catalog_url` corrections ever had (`billing.charge_for_action`), and nothing replaced it: a caller could loop `POST /cards/{card_id}/corrections` with `field_name: "catalog_url"` indefinitely, each call enqueuing a real, billed-to-DASHR Apify actor run — even resubmitting the identical URL. Fixed two ways: (1) a **no-op guard**, applied to *every* `field_name`, rejects (`NoOpCorrectionError`, 400) a correction whose `corrected_value` equals the field's current value before writing anything — this alone closes both the trivial "resubmit the same catalog_url" loop and a parallel "resubmit the same value repeatedly" free-rescore-spam loop on any other field; (2) a **per-user cooldown** (`_CATALOG_URL_CORRECTION_COOLDOWN`, 1 minute) on `catalog_url` corrections specifically (`CorrectionRateLimitedError`, 429), since the no-op guard alone doesn't stop cycling between two genuinely different real URLs. Explicitly a cheap first line of defense, not a comprehensive rate limiter — documented as such in code.
- **Quality — deduplication.** The 11-field supplier-profile list and the `marketplace_vintage_years` derivation, previously copy-pasted between `run_all_signal_lookups` and `rerun_supplier_profile_lookup`, are now a shared `_SUPPLIER_PROFILE_FIELDS` constant and `_apply_marketplace_vintage_years` helper in `enrichment_service.py`. `_has_correction_since_score`, previously hand-duplicated in `card_service.py` and `scoring_processing.py` (to dodge a `card_service` ↔ `scoring_processing` import cycle), is now a single `field_correction_service.has_correction_since_score` in a new dependency-free module both files import.
- **Quality — readability.** `correct_card_field` (previously one ~115-line, 4-branch function) is now a thin dispatcher over `_correct_text_field`/`_correct_company_name`/`_correct_email_or_phone`/`_correct_catalog_url` helpers, each independently readable and each owning its field type's no-op check.
- **Quality — test coverage.** `apps/web/__tests__/20-field-correction.test.tsx` (new) covers `InlineEditableValue`'s happy path (edit → save → corrected value displayed, right request body) and error path (a 400 renders inline, draft input preserved, not silently reverted) — previously the correction UI itself had zero frontend test coverage (only the Phase 2 rescore CTA did).
- **Polish.** Corrected text-field values are now `.strip()`'d server-side (parity with the frontend's own `.trim()` and with how extraction normalizes these same columns); a code comment clarifies why `catalog_url`'s "vanished Company row" case intentionally stays `CardHasNoCompanyError` rather than mirroring `enrich_company_now`'s `CompanyNotEligibleForEnrichmentError` (different condition — that one covers "exists but not pending," which doesn't apply here); a long import line was wrapped.
- **Investigated, no change needed.** The reviewer also flagged CSV/formula-injection risk on corrected values reaching `POST /cards/export` — already fully mitigated by `export_service.py`'s pre-existing `_csv_safe`/`_FORMULA_PREFIXES` guard, which applies uniformly to every exported field regardless of whether it originated from OCR or a correction. Verified, not a real gap; no code changed.

## API endpoints (apps/api)
- `POST /cards/{card_id}/corrections` — org-authenticated (visibility via `get_visible_card`, same admin/own-org/own-user scoping as every other card endpoint) — body `CardFieldCorrectionRequest { field_name: "full_name" | "job_title" | "address" | "products_offered" | "company_name" | "email" | "phone" | "catalog_url", corrected_value: str (1-2000 chars), record_id: UUID | None }`. `record_id` is **required** when `field_name` is `"email"`/`"phone"` (identifies which `CardEmail.email_id`/`CardPhone.phone_id` row on this card is being corrected) and **must be omitted/null** for every other `field_name`. Writes one `FieldCorrection` row, applies the corrected value to the live field, and — only for `catalog_url` — re-enqueues the IndiaMART supplier-profile re-fetch (never billed, see Phase 2). Response: refreshed `CardDetailOut` (same shape `GET /cards/{card_id}` already returns), now also carrying `rescore_available: bool` (Phase 2) — `true` once this correction postdates the card's last score.
  - `404` — card not visible/not found (`CardNotFoundError`), or `record_id` doesn't match a `CardEmail`/`CardPhone` row on this card (`FieldCorrectionRecordNotFoundError`)
  - `400` — `corrected_value` fails the same validator/normalizer the extraction pipeline already uses for that field type — malformed email, unparseable phone, non-`indiamart.com` URL for `catalog_url` (`InvalidCorrectionValueError`); or `field_name == "catalog_url"` but the card has no linked company yet (`CardHasNoCompanyError`, same exception `enrich-company` already raises)
- **Phase 2 (done)**: `POST /cards/{card_id}/score` (from `10-lead-scoring`, unchanged route) gains one new eligibility branch — `card.lead_score is not None` no longer unconditionally raises `CardAlreadyScoredError`; it's only raised when no `FieldCorrection` postdates `scored_at`. When one does, the rescore proceeds and is free (no `billing.charge_for_action` call at all — distinct from the free-*allowance* tier, which still increments a usage counter). `CardOut`/`CardDetailOut` both gain `rescore_available: bool` (default `false`) for this — `list_cards` deliberately leaves it `false` on every row rather than computing it per-row (would fan out into one extra query per card on every page load); only the single-card paths (`to_card_out`, `get_card_detail`) compute a real value.

## Database changes
- New table `field_corrections` (org-scoped where possible, per the org_id-everywhere rule — `org_id` is nullable only because `User.org_id` itself is nullable, mirroring that column, not a deliberate exemption):
  - `correction_id` UUID PK, `gen_random_uuid()`
  - `org_id` UUID, FK → `organizations.org_id`, nullable
  - `card_id` UUID, FK → `visiting_cards.card_id`, `ondelete=CASCADE`, not null
  - `corrected_by_user_id` UUID, FK → `users.user_id`, not null
  - `field_name` String, not null (`"full_name" | "job_title" | "address" | "products_offered" | "company_name" | "email" | "phone" | "catalog_url"`)
  - `record_id` UUID, nullable — the `CardEmail.email_id`/`CardPhone.phone_id` this correction targeted, when `field_name` is `"email"`/`"phone"`; null otherwise
  - `original_value` Text, nullable (null when there was nothing to correct from, e.g. a card with no company being given its first `company_name`)
  - `corrected_value` Text, not null
  - `created_at` TIMESTAMPTZ, `server_default=now()`
  - Indexes: `org_id`, `card_id`, `field_name` (accuracy reporting will filter/group by all three)
  - Append-only — never updated or deleted, mirroring `WalletTransaction`/`CompanyEnrichment`'s existing ledger/audit convention
- `apps/api/schemas/cards.py` — `CardEmailOut` gains `email_id: uuid.UUID`, `CardPhoneOut` gains `phone_id: uuid.UUID` (both additive, no migration needed — `CardEmail.email_id`/`CardPhone.phone_id` already exist as PKs, just not previously serialized).
- No changes to `VisitingCard`, `Company`, or `CompanySignals` columns — every field this step corrects already exists on one of those three tables.

## Rules for implementation
- Every query on `field_corrections` filters by `org_id` where the caller's own org scoping applies (accuracy-report queries, once built, must never cross tenant boundaries) — the correction endpoint itself is authorized purely through the existing `get_visible_card` card-visibility check, matching how every other card-mutating endpoint (`enrich-company`, `score`, `reprocess`) is already scoped.
- No raw SQL string interpolation — SQLAlchemy query builder or bound params only.
- Business logic lives in `card_service.py`/`enrichment_service.py`, never in the router.
- **A `FieldCorrection` row must be written, with both `original_value` and `corrected_value` populated, in the same DB transaction as the live field mutation it describes** — never mutate a field and skip the audit row, and never write the audit row without applying the correction.
- **Correcting `company_name` must never rename the shared `Company` row in place.** Reuse `extraction_service._get_or_create_company` to re-match/dedupe against the corrected name (reusing an existing `Company` row with that normalized name, or creating a new one) and repoint `card.company_id` — `Company` is a cross-org cache, and an in-place rename would silently relabel that company for every other org sharing the cached row.
- **Correcting `catalog_url` is the one field-level exception to that rule** — `CompanySignals.catalog_url` is itself shared, non-org-scoped cache data (per `19-data-enrichment-indiamart`), so the correction updates it in place on the shared row, benefiting every org matched to that same company, and must re-enqueue `rerun_indiamart_supplier_profile_task` to refresh the `indiamart_*` fields — never leave stale supplier-profile data attached to a corrected URL.
- **No `field_name` is ever billed (Phase 2).** Correcting `catalog_url` never calls `billing.charge_for_action` at all — it succeeds even at 0 wallet balance with the free `"enrichment"` allowance exhausted, since it's fixing a mistake in an already-paid-for enrichment, not a new billable action. `billing.ACTION_TYPES` stays a closed `("parse", "enrichment", "scoring")` tuple; correction was never, and still isn't, a fourth action type. Since no charge is ever made, `rerun_indiamart_supplier_profile_task` must never call `billing.refund_action` on a retry-exhaustion failure either — `refund_action` unconditionally decrements `FreeActionAllowance.used_count`, which would be wrong for a count that was never incremented.
- **A correction that postdates a card's last score unlocks exactly one free rescore (Phase 2).** `score_card_now` (`10-lead-scoring`) still raises `CardAlreadyScoredError` when `lead_score is not None`, *unless* a `FieldCorrection` row's `created_at` is after `scored_at` — in that case the rescore proceeds and skips `billing.charge_for_action` entirely (not merely "covered by the free allowance" — genuinely never charged, and the allowance counter is never touched). `score_card_task` independently re-derives the same eligibility (`is_rescore = card.lead_score is not None`, then re-checks the correction-postdates-score condition) rather than trusting the enqueue-time decision, matching this task's existing re-check-everything idempotency philosophy — two racing enqueues can't both slip a rescore through, and a stuck/retried task re-validates on every attempt. `billing.refund_action` must never be called on a free rescore's retry-exhaustion failure, same reasoning as `catalog_url`'s re-fetch.
- **A corrected value goes through the same validator/normalizer the extraction pipeline already uses for that field type** before being persisted — `email` through `extraction_service._normalize_email`, `phone` through `_normalize_phone`, `company_name` through `_normalize_company_name`/`_get_or_create_company`, `catalog_url` through `_is_indiamart_url`. A value that fails validation raises `InvalidCorrectionValueError` (400) and nothing is written — a human-typed correction is not automatically trustworthy input either.
- `record_id` is required and must resolve to a `CardEmail`/`CardPhone` row belonging to this exact `card_id` when `field_name` is `"email"`/`"phone"` (else `FieldCorrectionRecordNotFoundError`, 404), and must be `None` for every other `field_name` (else a 422 validation error at the Pydantic layer).
- The `rerun_indiamart_supplier_profile_task` Celery task loads `Company`/`VisitingCard` fresh inside the worker via `SessionLocal()` — never trust state passed across the Celery broker beyond the plain-string ids/URL already validated at the API layer.
- `get_local_presence_provider()` must keep returning the stub in `settings.environment == "test"` for this re-fetch path too — the test suite must never make a real, billed Apify call, same rule `19-data-enrichment-indiamart` already established.
- **(Phase 4) Every correction must reject a no-op (`corrected_value` equal to the field's current value) before writing anything.** Since no correction is billed, an identical resubmission would otherwise be a free way to repeatedly unlock a rescore (any field) or re-trigger a paid Apify re-fetch (`catalog_url`) for zero actual change — `NoOpCorrectionError` (400), checked per-branch in `_correct_text_field`/`_correct_company_name`/`_correct_email_or_phone`/`_correct_catalog_url` right after each resolves the field's current value, before any mutation or `FieldCorrection` write.
- **(Phase 4) `catalog_url` corrections additionally carry a 1-minute per-user cooldown** (`_catalog_url_correction_on_cooldown`, `CorrectionRateLimitedError`, 429) — the no-op guard alone doesn't stop a script cycling between two genuinely different real URLs, and this is the one correction type with a real per-call financial cost now that it's never billed. Explicitly documented as a cheap first line of defense, not a comprehensive rate limiter — do not treat this as "solved," and do not remove it under the assumption the no-op guard alone is sufficient.
- **(Phase 4) `field_correction_service.has_correction_since_score` lives in its own dependency-free module**, not inside `card_service.py` — `workers/scoring_processing.py` needs it too, and `card_service.py` already imports `score_card_task` from that worker module, so a `scoring_processing -> card_service` import would be circular. Any future helper needed by both modules belongs in a similarly dependency-free location, not duplicated by hand.

## Definition of done
- [ ] `field_corrections` table exists at migration head `0021`, cascades away with its parent `VisitingCard` (`ondelete=CASCADE`), append-only (no update/delete path anywhere in the codebase)
- [ ] Correcting `full_name`/`job_title`/`address`/`products_offered` writes a `FieldCorrection` row with the pre-correction value as `original_value` and updates the live `VisitingCard` column; a second correction of the same field writes a second row rather than overwriting the first
