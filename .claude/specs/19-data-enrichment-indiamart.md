# Spec: Data Enrichment — IndiaMART

## Overview
This is a focused follow-up to `07-data-enrichment`'s local-presence signal (item 11 of that step's eleven-source fan-out), split into its own spec because IndiaMART enrichment turned out to have enough scope of its own to track separately from the other ten sources. `07-data-enrichment` shipped `LocalPresenceProvider.lookup_marketplace` as a stub returning "no signal found" — this step replaces that stub with a real implementation, delivered in two phases:

- **Phase 1 (done): `catalog_url` only.** A real `ApifyLocalPresenceProvider` backed by Apify's `apify/google-search-scraper` actor Googles `"{company name}" IndiaMart`, cross-checks candidate results against the queried company to reject fuzzy/coincidental matches, and — through a fallback cascade (company name, combined with one product and phrased to target the catalogue page directly when the card captured one → email domain → the card's own website → that website's legal name via its Terms/Privacy page → legal/company name plus that same product again, plainer phrasing) — resolves the company's public IndiaMART storefront/catalogue URL. The company-name-plus-product combination was promoted to the first-priority attempt (rather than the original last-resort position) since it more reliably ranks the seller's own catalogue page above unrelated results than a bare `"{name}" IndiaMart` query.
- **Phase 2 (done): supplier-profile details via a second actor.** A second, independent Apify actor — the real "IndiaMart Scraper" (id `g74tjdx6IthmoJoyO`, confirmed live against Apify's public actor-metadata API, not guessed), called in `mode: "supplierProfile"` directly against the Phase-1 `catalog_url` — pulls the seller's own IndiaMART profile-page data: rating, rating count, member-since year, business type, employee-count band, annual-turnover band, year established, GST number, GST registration year, and call-response rate. This call also closes Phase 1's gap: `isVerifiedExporter` → `marketplace_verified_badge`, and `memberSince` → a derived `marketplace_vintage_years` (`current_year - member_since_year`). **`marketplace_located_in_industrial_area` stays unfilled** — this actor's confirmed `supplierProfile` output has no full postal address (only city/state), and no other source in this spec provides one to pattern-match against for industrial-estate markers (MIDC/GIDC/etc.); this is a genuine, documented gap, not an oversight.
- **Phase 3 (done): live-testing fixes.** Manually re-enriching real companies (not stubbed/mocked) surfaced three gaps between the actor's *declared* output schema (used to design Phase 2) and its *actual* live behavior:
  1. **`memberSince` format.** Confirmed live (a real supplier, "AGGARWAL ENTERPRISES") that this field can be a tenure duration ("3 yrs") rather than a 4-digit join year ("2015") as the actor's own schema description implied — the original parser only handled the year format and silently left `indiamart_member_since_year`/`marketplace_vintage_years` `None` for the duration format. `_parse_member_since_year` now handles both: a direct year is used as-is; a duration is converted via `current_year - N`.
  2. **`gstRegistrationDate`/`callResponseRate` are real, not absent.** The same live call returned real values for both (`"2017"`, `"84%"`) — fields the actor's *declared* schema omitted, which is why Phase 2 shipped them as permanent-null placeholders. Now unlocked as real fields: `callResponseRate` maps directly to `indiamart_call_response_rate` (String, unchanged). `gstRegistrationDate` has only ever been observed as a bare year, never a full date, so the column was renamed/retyped from `indiamart_gst_registration_date` (Date) to **`indiamart_gst_registration_year`** (Integer, migration `0020`) rather than fabricating a Jan-1 calendar date out of information not actually known.
  3. **`gstNumber` can be a badge label, not a GSTIN.** The same live call returned `gstNumber: "TrustSEAL"` — an IndiaMART badge label, not a real GSTIN, for a supplier whose page doesn't publicly expose one. `_validate_gstin` now shape-checks the value (the standard 15-character GSTIN format) before it's ever stored/displayed; a non-matching value becomes `None` instead of a confusing "GST TrustSEAL" badge.
  4. **`catalog_url` domain hardening + phrasing.** The same ambiguous, very-common company name ("AGGARWAL ENTERPRISES", no email/website on the card to disambiguate) once resolved to `trustseal.indiamart.com/members/...` — a real, relevant IndiaMART page, but a "TrustSEAL member" verification page, not a storefront, which the Phase-2 supplier-profile actor call can't scrape (confirmed live: zero rows back). `_pick_best_indiamart_url` now ranks candidate URLs by storefront-profile-over-proddetail (unchanged) and, as a new tie-breaker, the standard `www.indiamart.com`/`indiamart.com` domain over any other `indiamart.com` subdomain — used by both `_extract_catalog_url` (Google-search results) and `_find_indiamart_link_on_page` (website-scrape fallback). Step 1's query phrasing was also strengthened from `"{name}" {product} IndiaMart catalogue` to **`"{name}" {product} IndiaMart Catalogue Profile`**, to more directly target the actual profile page.
- **Phase 4 (done): further disambiguation + a code-review pass.** Continued live testing against "AGGARWAL ENTERPRISES" (still resolving to the wrong same-named company even after Phase 3's domain hardening) surfaced two more real gaps, followed by a full security/quality review of the whole feature:
  1. **`_pick_one_product` was picking a business-type word, not the product.** This card's `products_offered` reads "Manufacturer, Importers, Traders of Toys" — the original comma-split logic picked "Manufacturer" (the first fragment), so step 1's query never actually included "Toys", the one term that could have disambiguated this company from every other same-named "Aggarwal Enterprises". Fixed with two independent safeguards: text after the last standalone "of" is used when present (peeling off the business-type preamble entirely), and every resulting part is checked against a `_GENERIC_BUSINESS_TYPE_WORDS` list (manufacturer/trader/importer/exporter/business/seller/dealer/distributor/wholesaler/retailer/supplier/vendor, singular and plural) and skipped if it's purely a business-type/role word — this second check also covers phrasing with no "of" at all (e.g. "Manufacturer, Trader, Toys").
  2. **City/address-based disambiguation.** Even with the right product term, Google can still return multiple genuinely different, equally name-relevant "Aggarwal Enterprises" companies in different cities (confirmed live: one run resolved to a Kanpur company when the card's own address was in Delhi). `_extract_city_tokens` parses every city named in the card's `address` (Indian postal addresses conventionally suffix the city with a hyphen and 6-digit PIN code, e.g. "Delhi-110006"), and `_pick_best_indiamart_url` gained a same-city tie-break — profile-over-proddetail first (unchanged), then same-city-over-different-city, then standard-domain-over-subdomain (Phase 3, unchanged). `lookup_marketplace` gained an `address` parameter threaded through every cascade step. This narrows disambiguation but doesn't fully solve it for an extremely generic name with no other anchor (website/email domain) — documented as a known, accepted limitation, not chased further in this pass.
  3. **Code-review hardening** (`/code-review-feature`, full report not reproduced here): the Apify token moved from a URL query param to an `Authorization: Bearer` header in both actor calls — a query-param token would otherwise appear in `httpx.HTTPStatusError`'s stringified request URL, which `_run_lookup`'s broad exception logging would write to application logs in plaintext on any Apify 4xx/5xx (a real, routinely-triggered secret-exposure risk, not a hypothetical). `local_presence_provider.py` (829 lines after Phase 3) was split: URL-relevance/ranking helpers (`_looks_like_same_company_by_url`, `_is_indiamart_*`, `_pick_best_indiamart_url`, `_extract_city_tokens`) and the website-scrape fallback helpers (`_extract_links`, `_find_policy_page_urls`, `_extract_legal_name`, `_find_legal_name_via_policy_pages`) moved to a new sibling module, `local_presence_url_utils.py`; free-text field parsers (`_pick_one_product`, `_parse_member_since_year`, `_parse_year`, `_validate_gstin`) and the provider classes/dataclasses stayed. `_parse_member_since_year` now delegates its 4-digit-year branch to `_parse_year` instead of duplicating it. `_extract_city_tokens` dedupes repeated city names. `card_service.get_card_detail`'s 19-line hand-copied `"field": signals.field if signals else None` block collapsed into a `_COMPANY_SIGNAL_FIELDS` tuple + one dict-comprehension loop.

**This is a deliberate, scoped exception to `07-data-enrichment`'s "no paid or commercial data API" constraint**, which was written when Zauba Corp/MCA/GeM/LinkedIn/Google Maps were all blocked by bot protection and no paid workaround was in scope. Apify is a paid scraping-infrastructure provider (billed per actor run) — used here because IndiaMART's own search proved too keyword-fuzzy to trust directly (see the `thirdwatch/indiamart-supplier-scraper` incident below), and Google's own relevance ranking, accessed through Apify's search-scraper actor, is what actually finds the right storefront reliably. This exception is scoped to the IndiaMART signal only; it does not reopen the other ten sources in `07-data-enrichment` to paid alternatives.

## Rules for implementation
- Every query on `Company`/`CompanySignals` stays `company_id`-scoped only, no `org_id` — unchanged from `07-data-enrichment`, this step adds a column, not a new table.
- No raw SQL string interpolation — SQLAlchemy query builder or bound params only (`_card_email_domain`'s query uses the ORM, not a raw string).
- All Apify-calling and relevance-scoring logic lives in `enrichment_providers/local_presence_provider.py`; `enrichment_service.py` only threads inputs through and maps the result onto `company_signals` columns — the router/worker layers stay thin, per `07-data-enrichment`'s existing rule.
- **Never trust a fuzzy keyword match as `catalog_url`.** Two production incidents drove `_looks_like_same_company_by_url`'s design and must keep passing in `test_local_presence_provider.py`: (1) querying the superseded `thirdwatch/indiamart-supplier-scraper` actor directly with a full company name once returned an unrelated cosmetics supplier as the top "match"; (2) even after switching to Google-search-plus-domain-filtering, querying "DASHR" matched an unrelated supplier purely because their listing's *description* mentioned an unrelated product literally named "Dashr Timing Gates". The check is **URL-slug-based** (position of the queried company's significant words within the candidate URL's path, not the result's title/description text) for exactly this reason — title/description proved too easily coincidentally keyword-matched.
- The relevance check is **position-based, not length-ratio-based** — a short company name legitimately leading a long, legally-suffixed slug (e.g. `"DASHR"` leading `"dashrmaterialhandlingsolutions-opc"`) must be accepted, while the same short word merely appearing deep inside an unrelated slug must be rejected. Do not regress this back to a length-ratio check; see `_MAX_MATCH_START_OFFSET`'s docstring in `local_presence_provider.py` for the incident that ruled it out.
- The fallback cascade (name-plus-one-product targeting the catalogue page → email domain → website's direct IndiaMART link → website's legal name via policy page → legal/name-plus-that-same-product again, plainer phrasing) only advances a step when the prior one comes up genuinely empty after its own retry — each step is a real, billed Apify call or HTTP fetch, never spend one proactively once an earlier step already succeeded.
- Step 1's company name stays the sole exact-phrase-quoted term regardless of whether a product is combined in — the product/`"IndiaMart Catalogue Profile"` wording is appended unquoted after it (via `_search`'s `suffix` parameter), never folded into the quoted phrase itself, since quoting "name plus product" together as one exact phrase would rarely match anything on Google.
- **(Phase 3)** Whenever multiple relevant candidate URLs are available, `_pick_best_indiamart_url` must be used to choose among them (never just `[0]` off an unranked list) — storefront-profile over proddetail first, then the standard `www.indiamart.com`/`indiamart.com` domain over any other `indiamart.com` subdomain. A `trustseal.indiamart.com` (or similar) URL can pass every relevance check yet still be unscrapable by `lookup_supplier_profile`'s actor — confirmed live, not assumed.
- **(Phase 3)** `_parse_member_since_year` must handle both a direct 4-digit year and an "N yrs"/"N years" tenure duration — never assume the actor's declared schema description ("Year the supplier joined IndiaMart") matches its actual live output; both formats are confirmed real.
- **(Phase 3)** `indiamart_gst_number` must go through `_validate_gstin`'s shape check before being stored — never trust `gstNumber` verbatim, since it's confirmed live to sometimes carry a badge label (e.g. "TrustSEAL") instead of an actual GSTIN.
- **(Phase 3)** Never fabricate precision that isn't there — `indiamart_gst_registration_year` is an int (a bare year is all that's ever been observed), not a `Date` with a made-up month/day.
- **(Phase 4)** `_pick_one_product` must never return a business-type/role word (manufacturer, trader, importer, exporter, business, seller, dealer, distributor, wholesaler, retailer, supplier, vendor) as if it were the product — check both the "of"-split and the `_GENERIC_BUSINESS_TYPE_WORDS` filter keep passing; a card's `products_offered` is commonly phrased "Manufacturer, Trader of X" or "Manufacturer, Trader, X" and only "X" is the actual product to search on.
- **(Phase 4)** The Apify token is sent via an `Authorization: Bearer` header in both `_search` and `lookup_supplier_profile`, never a URL query param — a query-param token ends up in `httpx.HTTPStatusError`'s stringified request URL, which `_run_lookup`'s broad exception logging would otherwise write to application logs in plaintext on any Apify 4xx/5xx. Do not regress this back to `params={"token": ...}`.
- **(Phase 4)** `local_presence_url_utils.py` holds URL-relevance/ranking and website-scrape-fallback helpers; `local_presence_provider.py` holds free-text field parsers (next to the dataclasses they populate) and the provider classes. New helpers belong in whichever module matches their concern — don't let `local_presence_provider.py` regrow past this split.
- **(Phase 4)** City/address disambiguation narrows, but does not fully solve, ambiguity for an extremely generic company name with no other anchor (website/email domain) — a same-city collision (multiple same-named companies in one city) is a documented, accepted limitation, not a bug to keep chasing.
- Generic/free email providers (Gmail, Yahoo, Outlook, etc. — see `_GENERIC_EMAIL_DOMAINS`) are never used for the email-domain fallback step — a domain like `gmail.com` can't identify any one company's storefront.
- `website_fetch.fetch_html` must stay SSRF-guarded (`is_safe_public_url`, checked before every redirect hop) — the card's `website` field is unvalidated vision-LLM OCR output from a user-uploaded image, not a trusted input.
- `get_local_presence_provider()` must keep returning the stub whenever `settings.environment == "test"`, regardless of whether an Apify token happens to be configured — the test suite must never make a real, billed Apify call. This covers both actors.
- API contract changes are the `CardCompanyOut` Pydantic model — the matching TS type in `lib/api.ts` is hand-aligned to match, never assumed.
- **(Phase 2)** Lookup #12 (`indiamart_supplier_profile`) is gated on lookup #11 having found a `catalog_url` in the same run — never spend this second billed Apify call chasing a URL that wasn't found.
- **(Phase 2)** `marketplace_vintage_years` is derived from `signals.indiamart_member_since_year` **after** the `for key, value in data.items(): setattr(signals, key, value)` loop in `run_all_signal_lookups`, reading the already-set `signals.` attribute — not written into the `data` dict, which is fully consumed by that loop and has no effect afterward. This is the same placement `hiring_signal`/`estimated_revenue_band` already use.
