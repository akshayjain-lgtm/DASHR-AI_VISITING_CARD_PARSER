# Spec: Wallet Usage

## Overview
This feature wires the "Billing" primitives from Step 14 (Wallet Recharge) into the actual work they're meant to pay for. It is the "Wallet usage" workflow stage in CLAUDE.md's core workflows list â€” distinct from "Billing" (recharge + invoicing) and coming after it. Today, `POST /cards/process`, `/cards/enrich-companies`, `/cards/score`, and their single-card counterparts (`/cards/{id}/reprocess`, `/cards/{id}/enrich-company`, `/cards/{id}/score`) enqueue Celery work completely free of charge â€” `billing.debit_wallet` exists but nothing calls it. This spec adds a per-user, per-action-type free allowance (20 free parses, 20 free enrichments, 20 free scorings, tracked independently), and once that allowance for an action type is exhausted, every further action of that type is charged at its **own, separate rate** â€” parsing, enrichment, and scoring are billed independently at the existing launch rates already seeded in Step 14's `pricing_rates` table (â‚ą5 per parse, â‚ą3 per enrichment, â‚ą2 per scoring), never a single blended per-action price. Once an action type's free allowance is used up, every action of that type is gated on a race-safe wallet balance check against *that type's own rate* â€” blocking the action outright (never running OCR/enrichment/scoring, never enqueuing the Celery task) at a zero or insufficient balance.

## Depends on
- Step 14 (Wallet Recharge) â€” this spec extends `billing.py`, `Wallet`, `WalletTransaction`, and `PricingRate`, all built there. It only adds new functions/columns; nothing from Step 14 is removed or renamed.
- Step 05 (Parsing Visiting Card), Step 07 (Data Enrichment), Step 10 (Lead Scoring) â€” this spec wires charging into the endpoints those steps built (`card_service.reprocess_card`/`enrich_company_now`/`score_card_now`/`enqueue_processing`/`enqueue_enrichment`/`enqueue_scoring`); it does not change their eligibility rules, only adds a charge step ahead of enqueue.

## API endpoints (apps/api)

No new endpoints. Existing endpoints change response shape and/or gain a new error status:

- `POST /cards/process` â€” response gains `wallet_blocked_count: int` alongside the existing `enqueued_count`. Cards matched but blocked by a zero/insufficient balance are no longer enqueued and are counted here instead.
- `POST /cards/enrich-companies` â€” response gains `wallet_blocked_count: int` alongside existing `enqueued_count`/`skipped_count` (a distinct bucket from `skipped_count`, which remains ineligibility-only: no linked company, company not pending, duplicate company in the same batch).
- `POST /cards/score` â€” same shape change as `/cards/enrich-companies`.
- `POST /cards/{card_id}/reprocess` â€” now returns `402 Payment Required` (`detail: str`) if the user's free parse allowance is exhausted and their wallet balance can't cover the **parse rate (â‚ą5)**. Card status is left unchanged (still `"failed"`) â€” a blocked reprocess must not flip it to `"new"`.
- `POST /cards/{card_id}/enrich-company` â€” now returns `402 Payment Required` under the same condition, checked against the **enrichment rate (â‚ą3)** â€” a separate rate from parse, looked up independently.
- `POST /cards/{card_id}/score` â€” now returns `402 Payment Required` under the same condition, checked against the **scoring rate (â‚ą2)** â€” again its own independent rate, not shared with parse or enrichment.
- `GET /wallet` â€” response gains `free_actions_remaining: {parse: int, enrichment: int, scoring: int}`, each `max(free_limit - used_count, 0)` for that action type. Each action type keeps its own free-limit and its own rate, so this map (and the pricing behind it) can never be collapsed into a single number.

## Database changes

One new Alembic migration `0012_wallet_usage.py` (revises `0011_seller_profile_gst_billing` â€” `0011` was already taken by that spec's migration by the time this one landed, so the spec's originally-stated `0011_wallet_usage.py` was renumbered `0012` during implementation):

- **Add column** `pricing_rates.free_limit` (Integer, not null, `server_default="20"`) â€” the free-action cap for that action type, versioned alongside `rate_inr` on the same row (a rate change and a free-limit change both become a new `pricing_rates` row via the existing `effective_from` mechanism; no separate config table needed). Backfills existing 3 seeded rows to `free_limit=20` each â€” `pricing_rates` already carries a separate row per `action_type` (`parse` @ â‚ą5, `enrichment` @ â‚ą3, `scoring` @ â‚ą2 from Step 14's seed), so `free_limit` and `rate_inr` are both looked up per action type from that same row; nothing here introduces a shared/blended rate across the three.

- **New table `free_action_allowances`** â€” one row per `(user_id, action_type)`, User-scoped like `wallets` (no `org_id` â€” matches CLAUDE.md's explicit carve-out for Wallet/WalletTransaction):
  - `free_action_allowance_id` (UUID, PK, `gen_random_uuid()`)
  - `user_id` (UUID, FK â†’ `users.user_id`, not null)
  - `action_type` (String, not null â€” `"parse"` | `"enrichment"` | `"scoring"`)
  - `used_count` (Integer, not null, `server_default="0"`) â€” lifetime count of actions of this type by this user, free or paid; only ever incremented, never reset or decremented
  - `created_at` (TIMESTAMPTZ, not null, `server_default=now()`)
  - `updated_at` (TIMESTAMPTZ, not null, `server_default=now()`, `onupdate=now()`)
  - Unique index on `(user_id, action_type)` â€” one counter per user per action type, created lazily on first use (mirrors `wallets`' one-row-per-user pattern).

New model file `apps/api/app/models/free_action_allowance.py` (`FreeActionAllowance`), registered in `apps/api/app/db/base.py`'s metadata import list. `PricingRate` model (`apps/api/app/models/pricing_rate.py`) gains the `free_limit: Mapped[int]` column.

## Background jobs

No new Celery tasks, but the three existing ones (`process_card`, `enrich_company_task`, `score_card_task`) each gain a `billed: bool = False` parameter and a refund call on permanent failure â€” see the 3rd-round "Implementation note" below. The charge itself still happens synchronously ahead of `.delay()`, at the same point CLAUDE.md requires the balance check to happen ("ahead of the OCR/enrichment/scoring call, not after"); what changed is what happens if the enqueued work then genuinely fails.

## Rules for implementation

- Every query on `free_action_allowances` filters by `user_id` (not `org_id` â€” User-scoped like `wallets`, per CLAUDE.md)
- No raw SQL string interpolation â€” SQLAlchemy query builder or bound params only
- Business logic for charging lives in `services/billing.py` (`charge_for_action`) and is called from `services/card_service.py`; `routers/cards.py` only maps `InsufficientBalanceError` to a 402 â€” no billing logic in the router
- `charge_for_action(db, user_id, action_type, reference_id=None) -> bool` (returns `True` if billed from the wallet, `False` if covered by the free allowance) is the **only** entry point card_service calls for this â€” it must, in one sequence sharing a single row lock per resource:
  1. Lock (or lazily create) the `(user_id, action_type)` `FreeActionAllowance` row (`SELECT ... FOR UPDATE`)
  2. Read the currently-effective `PricingRate` row for `action_type` (rate + `free_limit`), same lookup pattern as `get_current_rate`
  3. If `used_count < free_limit`: increment `used_count`, commit, return `False` â€” no wallet touched, no `WalletTransaction` row (free actions aren't ledger events, since no money moves)
  4. Else: lock the wallet row, and if `balance_inr < rate_inr`, raise `InsufficientBalanceError` **without incrementing `used_count`** (the action never happened â€” nothing to count); otherwise debit the wallet, write the `WalletTransaction` row (`transaction_type=f"{action_type}_debit"`, `reference_id=<card_id>`), increment `used_count`, and commit **all of it â€” allowance increment, balance decrement, and ledger insert â€” in a single transaction**. Do not call the existing `debit_wallet` as a separately-committed step from here: it commits internally, and if the allowance increment failed afterward, the wallet would be debited with no matching allowance update. Inline the equivalent of `debit_wallet`'s body instead, sharing `_lock_or_create_wallet`, so the whole charge is one atomic commit.
- `charge_for_action` must run, and either succeed or raise, **before** any Celery `.delay()` call for that action, and before any DB mutation to the card's own state that a subsequent failure couldn't cleanly undo (e.g. `reprocess_card` must call `charge_for_action` before flipping `card.status` back to `"new"`, not after â€” a blocked reprocess must leave the card exactly as it was, still `"failed"` and still eligible for a real retry once funded)
- In `enqueue_processing`/`enqueue_enrichment`/`enqueue_scoring`, the whole eligible batch is charged in one call to `billing.charge_for_bulk_action(db, user_id, action_type, count, reference_id=...)` (see "Implementation note" below) rather than one `charge_for_action` call per card â€” it never raises, returning how many of `count` were actually chargeable (free + paid); the caller enqueues Celery work for only that many (in whatever order the eligible list is in) and reports the remainder as `wallet_blocked_count`, without aborting the rest of the batch
- In the three single-card endpoints (`reprocess_card`, `enrich_company_now`, `score_card_now`), `InsufficientBalanceError` propagates up to the router uncaught by the service, and `routers/cards.py` maps it to `402 Payment Required`
- `PricingRate.free_limit`, like `rate_inr`, is read through `billing.py`, never hardcoded â€” same "configurable data, not a branch" rule CLAUDE.md applies to scoring weights and per-action prices
- Parse, enrichment, and scoring are priced and rate-limited **independently** â€” `charge_for_action` always looks up `rate_inr`/`free_limit` for the specific `action_type` passed in, never a single shared constant; a rate or free-limit change for one action type (e.g. raising the parse price) must have zero effect on the other two
- `FreeActionAllowance.used_count` increments on every parse/enrich/score by that user, whether free or wallet-debited (CLAUDE.md) â€” never reset, never decremented, and never shared across action types (a `(user_id, action_type)` pair is its own independent counter)
- API contracts are Pydantic models in `apps/api/app/schemas/`; TS types in `apps/web/lib/api.ts` are hand-written to match (no codegen step in this repo yet, same as every existing type in `api.ts`)

**Implementation note (post-review):** bulk parse/enrich/score debits are consolidated into a single collective `WalletTransaction` per batch instead of one row per card, so a large batch's transaction history stays readable â€” a design change from this spec's original per-card `charge_for_action` loop, made after initial implementation. This adds:
- `wallet_transactions.quantity` (Integer, `server_default="1"`), added by a follow-up migration `0013_wallet_transaction_quantity.py` â€” how many parse/enrich/score actions one row covers. 1 for every existing single-card debit and for recharge/adjustment rows; the batch size for a collective bulk row.
- `billing.charge_for_bulk_action(db, user_id, action_type, count, *, reference_id=None) -> int` â€” consumes the free allowance first (same as `charge_for_action`), then bills however many of the remaining `count` the wallet can actually afford as **one** row (`amount_inr = rate_inr * affordable`, `quantity = affordable`). Never raises; returns the number actually chargeable (free + paid) out of `count`, which may be less than `count`. `reference_id` is only ever stored on the row when exactly one action ends up chargeable (whether `count` was 1 or a bulk request happened to only afford one) â€” a genuinely collective row has no single card to point at, so it's `NULL` there, and `quantity` is what tells the reader "how many."
- `enqueue_processing`/`enqueue_enrichment`/`enqueue_scoring` build their full eligible-card list first (unchanged eligibility rules), then call `charge_for_bulk_action` once for the whole list, then enqueue Celery work for only the first N (however many were chargeable) â€” the remainder is `wallet_blocked_count`, exactly as before, just derived from one collective charge instead of N individual ones.
- `WalletTransactionOut`/the frontend `WalletTransactionOut` type both gain `quantity`; the wallet page's transaction row shows "on N cards" beneath the type badge when `quantity > 1`, and nothing extra for a single-card (`quantity == 1`) row.
- `charge_for_action` (the single-card entry point used by `reprocess_card`/`enrich_company_now`/`score_card_now`) is unchanged â€” it still writes its own one-row, quantity-1 (via the column's default), `reference_id`-carrying transaction per call, and still raises `InsufficientBalanceError` on a block. It is not reimplemented in terms of `charge_for_bulk_action`, to avoid touching its already-tested single-action contract (return `bool`, raise on block) for a function whose bulk counterpart has a different contract (return `int`, never raise).

**Implementation note (2nd post-review pass â€” `/code-review-feature` findings):** a security review of the above found two real money-correctness bugs and a handful of quality items, all fixed:

- **Duplicate `card_ids` in one request.** `enqueue_scoring` had no dedup on `card_ids` â€” a client sending the same id 3 times billed and enqueued it 3 times over. Fixed with a `seen_card_ids` guard (mirrors `enqueue_enrichment`'s pre-existing `seen_company_ids` dedup, which already incidentally covered this for enrichment; `enqueue_processing` was already safe via its SQL `IN` clause).
- **Concurrent duplicate requests double-charging one card.** The eligibility read (`card.status`) in `reprocess_card` was unlocked, so two concurrent reprocess requests for the same failed card could both pass the `status == "failed"` check and both get charged. Fixed for `reprocess_card` specifically: `get_visible_card` gained a `for_update: bool = False` param; `reprocess_card` locks the card row, stages the `status = "new"` flip *without committing*, then calls `charge_for_action` â€” whose own commit now persists both the flip and the charge atomically, and whose `InsufficientBalanceError` path rolls back both together. A concurrent second request blocks on the row lock until the first commits, then correctly re-reads `status == "new"` and is rejected (409), not double-charged. Verified with a real multi-threaded test (`test_reprocess_concurrent_duplicate_requests_charge_exactly_once`) and live over real concurrent HTTP requests.
  - This fix uncovered a subtler bug: `_get_or_create_wallet`/`_get_or_create_allowance`'s lazy-row-creation path called `db.commit()` immediately, which â€” now that `reprocess_card` stages an uncommitted mutation before calling `charge_for_action` â€” would silently commit that staged mutation early (before the balance check could roll it back on insufficient funds). Both functions gained a `commit: bool = True` parameter; `charge_for_action`/`charge_for_bulk_action` pass `commit=False` (using `db.flush()` instead, still populating server defaults via a same-transaction `db.refresh()`), so a first-ever wallet/allowance row's creation no longer ends the transaction before the charge itself decides to commit or roll back. All other callers (`credit_wallet`, `debit_wallet`, `get_wallet`, `refund_action`) keep the default `commit=True`.
  - **Known residual gap, explicitly not fixed**: the same race exists for `enqueue_processing`/`enrich_company_now`/`enqueue_enrichment`/`score_card_now`/`enqueue_scoring` â€” none of them have a state mutation available at this layer to fold into the charge the way `reprocess_card`'s status flip does (repurposing the Celery tasks' own `"processing"`/`"enriching"` intermediate statuses was considered and rejected â€” the tasks' own fresh-delivery guards require seeing `"new"`/`"pending"` respectively, so setting those early would silently no-op the real work). Closing this fully needs a new claim column, which is a schema decision left as a fast-follow rather than done silently here.
- **Charge succeeds but the Celery enqueue itself never reaches the broker.** New `billing.refund_action(db, user_id, action_type, *, billed, reference_id)` reverses exactly one `charge_for_action` charge (decrements the allowance, and credits the wallet back via a `{action_type}_refund` transaction if it was billed) â€” wired into the `except` blocks of `reprocess_card`/`enrich_company_now`/`score_card_now` when `.delay()` itself raises. The bulk `_charge_and_enqueue` helper does **not** get this treatment for its own enqueue-broker-failure case â€” see the 3rd-round note below for why bulk couldn't get a *full* version of this, only the task-level half.
- **Quality**: extracted the repeated charge-then-enqueue-subset sequence (`enqueue_processing`/`enqueue_enrichment`/`enqueue_scoring`) into a shared `card_service._charge_and_enqueue` helper; extracted `billing._lock_allowance_and_rate` shared by `charge_for_action`/`charge_for_bulk_action`; replaced `WalletOut.free_actions_remaining: dict[str, int]` with a precise `FreeActionsRemaining` Pydantic model; added a `billing.ACTION_TYPES = ("parse", "enrichment", "scoring")` constant so `routers/wallet.py` doesn't hand-maintain its own copy; added `max_length=settings.max_bulk_upload_files` to `CardProcessRequest.card_ids` for consistency with the other bulk request schemas, now that this endpoint drives a real charge.

**Implementation note (3rd round â€” refund on genuine task failure, and a frontend staleness fix):** the 2nd round's "no self-serve refund flow" reading of CLAUDE.md was explicitly overridden â€” a user must not stay charged for a parse/enrichment/scoring action that was attempted but never actually completed, including when the Celery task itself runs and permanently fails (not just when the `.delay()` enqueue call never reaches the broker). This adds:

- `process_card`/`enrich_company_task`/`score_card_task` each gain a `billed: bool = False` parameter, propagated from the charge that enqueued them (card_service always passes the real value; the default only matters for tests/manual invocations that bypass the normal enqueue path). On every one of their existing permanent-failure branches (`process_card`: `VisionApiError` after exhausted retries, `ExtractionValidationError`, or any other unexpected exception; `enrich_company_task`: exhausted retries; `score_card_task`: exhausted retries), the task now calls `billing.refund_action(db, user_id, action_type, billed=billed, reference_id=<card_id>)` right after finalizing its own failure state. A task that succeeds never touches billing at all â€” only the permanent-failure branches do.
- `enrich_company_task` gets its refund's `user_id` from `source_card.user_id` (`Company` itself has no `user_id` â€” enrichment data is shared across orgs) â€” if `source_card_id` was never passed (shouldn't happen via the normal enqueue path, but defensively handled), the refund is skipped with a warning log rather than crashing.
- `score_card_task` has no persistent "failed" status at all (scoring only ever sets `lead_score`/`scored_at` on success, per its existing idempotency-note docstring) â€” the refund transaction is the *only* durable trace a permanent scoring failure leaves in the data model today; this spec doesn't add a scoring-failure status, only the refund.
- To make this work for **bulk** batches too (where `charge_for_bulk_action` bills N cards as one collective row, so there's no per-card ledger entry to look up "was this card free or paid"), `charge_for_bulk_action`'s return type changed from `int` to `tuple[int, int]` = `(free_used, paid_used)`. `card_service._charge_and_enqueue` now computes each card's `billed` flag by its position in the chargeable subset (`index >= free_used`) and passes it into that card's own `.delay()` call â€” no schema change needed, since the billed/free split only ever needs to be known once, at enqueue time, not reconstructed later from the ledger.
- Frontend: `apps/web/app/upload/page.tsx` passed its `CardDetailDrawer` an `onChanged` callback that only called `refreshCards`, never `refreshWallet` â€” so a parse/enrich/score action triggered from the drawer (as opposed to the upload page's own row icons or bulk buttons) charged the wallet correctly but left the page's balance/free-actions indicator stale until a manual reload. Fixed by having `onChanged` call both.
- **Known residual gap, still not fixed**: `_charge_and_enqueue`'s own enqueue-broker-failure case (the `.delay()` call itself raising, not the task running and failing) still isn't refunded for bulk batches â€” unlike the single-card paths. This is a much rarer failure than a task actually running and failing, and doing it precisely for a partially-enqueued bulk batch is more involved than the straightforward per-card `billed` flag this round added; left as a further fast-follow, documented in `_charge_and_enqueue`'s own docstring.

## Definition of done

- [ ] `alembic upgrade head` adds `pricing_rates.free_limit` (backfilled to 20 on the three seeded rows) and creates `free_action_allowances`
- [ ] A user with a fresh (never-used) `FreeActionAllowance` for `"parse"` can call `POST /cards/{id}/reprocess` (or get enqueued via bulk `/cards/process`) with a `0` wallet balance and it succeeds, incrementing `used_count` to 1 and writing no `WalletTransaction`
- [ ] After 20 parses by the same user, the 21st parse at a `0` or insufficient wallet balance is blocked: `POST /cards/{id}/reprocess` returns `402` and leaves `card.status`/`wallets.balance_inr`/`free_action_allowances.used_count` all unchanged; the bulk `POST /cards/process` equivalent enqueues nothing for that card and reports it in `wallet_blocked_count`, not `enqueued_count`
- [ ] After 20 parses, the 21st parse with a wallet balance â‰Ą the current parse rate succeeds, debits the wallet by exactly the rate, writes one `WalletTransaction` (`transaction_type="parse_debit"`, `reference_id=<card_id>`, correct `balance_after_inr`), and increments `used_count` to 21 â€” all in a way that a mid-sequence failure can't leave the wallet debited without the allowance incremented (or vice versa)
- [ ] Exhausting the free parse allowance has no effect on the enrichment or scoring allowances for the same user (independent counters) â€” verified by exhausting one and confirming the other two still start free
- [ ] `GET /wallet` returns `free_actions_remaining` reflecting `used_count`/`free_limit` for all three action types correctly, including `0` (not negative) once exhausted
- [ ] Concurrent calls to `charge_for_action` for the same user/action_type (simulated with two threads/sessions against the same row) never allow the wallet to go negative or the allowance to double-count â€” the row lock serializes them
- [ ] A funded user bulk-parsing 3 cards past their free allowance writes exactly **one** `WalletTransaction` (`quantity=3`, `amount_inr` = 3Ă— the parse rate, `reference_id=NULL`), not three; if the wallet can only afford 2 of those 3, the batch still writes exactly one row (`quantity=2`) and reports the third in `wallet_blocked_count`
- [ ] `POST /cards/score` with the same `card_id` repeated N times in `card_ids` bills/enqueues it at most once, with the other N-1 occurrences counted in `skipped_count`
- [ ] Two concurrent `POST /cards/{id}/reprocess` requests for the same failed card result in exactly one 200 and one 409, and exactly one `parse_debit`/free-allowance-increment â€” never two charges for one card
- [ ] If `process_card.delay()`/`enrich_company_task.delay()`/`score_card_task.delay()` itself raises inside `reprocess_card`/`enrich_company_now`/`score_card_now`, the charge is reversed (`{action_type}_refund` transaction if billed, allowance decremented) rather than left charged for work that was never even queued
